청중이 가장 자주 물어보는 질문이다. 슬라이드 1장에 **트레이드오프 표** 로 정리:

| 결정 | 대안 | 선택 이유 | 트레이드오프 |
| --- | --- | --- | --- |
| Spring Boot | Express, Django | JDK 21 LTS, 팀 친숙도, 강타입 | 부트 시간 |
| Mustache (SSR) | React (SPA) | 3주 일정·SEO·세션 인증 단순화 | 인터랙션 한계 |
| MySQL | PostgreSQL | 팀 익숙도, 도커 이미지 작음 | JSON 기능 약함 |
| Redis 세션 | DB 세션 | 무중단 재시작 | 운영 부담 |
| 모듈러 모놀리스 | MSA | 6명 3주 — MSA 비용 과함 | 향후 분할 필요 시 부담 |

> 핵심 질문 대비: "왜 MSA 안 썼나요?" "왜 React 안 썼나요?" "왜 NoSQL 아닌가요?"

## 4. 기술적 도전 Deep Dive

### 4.1 도전 발표 포맷 (한 도전 = 한 슬라이드 + 1분)

```
1. 문제 (10초)   "동시 예약 시 슬롯이 중복 발생"
2. 원인 (15초)   "서비스 레이어에서 가용 슬롯 조회 후 INSERT 사이에 다른 요청이 같은 슬롯을 INSERT"
3. 시도 (15초)   "Optimistic Lock → 충돌 시 사용자 경험 저하"
4. 해결 (15초)   "DB 유니크 제약 + 비관적 락으로 슬롯 단위 직렬화"
5. 결과 (5초)    "동시 100건 테스트에서 중복 0건, 응답 P95 380ms 유지"
```

### 4.2 발표할 도전 후보 (3개 준비, 시간 보고 1~2개 사용)

- [ ] 동시성 (예약·재고·주문 중복)
- [ ] 외부 API 장애 대응 (폴백, 서킷브레이커)
- [ ] N+1 → fetch join / EntityGraph
- [ ] 인증·세션 (Redis 외부화)
- [ ] 대용량 조회 → 페이징·인덱스 최적화
- [ ] 6명 동시 작업 시 충돌 → 모듈 소유권 분리
- [ ] AI 에이전트 협업 정착 (AGENTS.md, 환각 체크)

### 4.3 코드 한 조각 보여주기

발표 중 **코드는 최대 1~2개 슬라이드에 짧게**. 너무 길면 청중이 못 따라온다.

```java
// Before — 동시성 문제
@Transactional
public Reservation createReservation(...) {
    if (slotRepo.findAvailable(slotId).isEmpty()) {
        throw new SlotUnavailableException();
    }
    return reservationRepo.save(...);  // 여기서 중복 발생
}

// After — 비관적 락
@Transactional
public Reservation createReservation(...) {
    Slot slot = slotRepo.findByIdForUpdate(slotId)  // SELECT ... FOR UPDATE
        .orElseThrow(SlotUnavailableException::new);
    slot.reserve();
    return reservationRepo.save(...);
}
```

### 6.1 아키텍처·기술 선택

1. **"왜 모놀리스인가요? MSA가 더 좋지 않나요?"**

2. **"왜 SSR인가요? React/Vue가 더 좋지 않나요?"**
   → SEO + 세션 인증 단순화 + 일정 단축. SPA가 필요한 부분({{기능명}})만 부분 도입.

3. **"왜 MySQL인가요? PostgreSQL은요?"**
   → 팀 친숙도 + 도커 이미지 + 클라우드 매니지드 비용. PG의 JSON·확장성은 현 요구사항에서 불필요.

4. **"왜 JPA인가요? MyBatis는요?"**
   → 도메인 모델 위주 설계. 복잡 쿼리는 QueryDSL 도입.

## 8. 발표 자료 체크리스트

### 8.1 슬라이드 구성 (15분 기준 ~12장)

1. 표지 — 프로젝트명 + 한 줄 가치
2. 문제 정의 (1장)
3. 해결책 한 장 요약 (1장)
4. 시스템 다이어그램 (1장)
5. 기술 스택 + 의사결정 표 (1장)
6. 데모 1 (라이브, 슬라이드 = 컨텍스트만)
7. 데모 2 (라이브)
8. 기술 도전 1 (Before/After 코드)
9. 기술 도전 2 (선택)
10. 결과·지표 (1장 — 숫자 위주)
11. 팀 + 회고 + AI 협업 (1장)
12. 마무리 — 링크 (Repo, 블로그, 연락처)

### 8.2 발표 자료 디자인 원칙

- 슬라이드당 핵심 메시지 **하나**
- 글자 24pt 이상 (뒤에서도 보이게)
- 코드 폰트는 monospace, 14pt 이상
- 색상 3가지 이내 (브랜드 + 강조 + 기본 텍스트)
- 다크 테마 권장 (눈 피로 적음)
