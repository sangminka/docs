> **Must Have (M)**: MVP에 반드시 포함. 없으면 시스템 의미 없음.
> **Should Have (S)**: 중요하지만 없어도 동작. W3 여유 시 포함.
> **Could Have (C)**: 있으면 좋음. Out-Scope 후보.
> **Won't Have (W)**: 이번 버전에서 제외. v1.1로 이관.

## 8. 인수 기준 (Acceptance Criteria)

> **인수 기준**: 해당 기능이 "완성됐다"고 판단하는 구체적이고 테스트 가능한 조건.
> 모든 Must Have 기능은 인수 기준이 있어야 한다.

### AUTH-01. 회원가입

```
Given: 사용자가 회원가입 화면에서 유효한 이메일과 비밀번호(8자+)를 입력했을 때
When: 가입하기 버튼을 클릭하면
Then:
  - DB에 회원 정보가 BCrypt 인코딩된 비밀번호로 저장된다
  - 로그인 화면으로 리디렉션된다
  - "가입 완료" 플래시 메시지가 표시된다

Given: 이미 가입된 이메일로 가입을 시도했을 때
When: 가입하기 버튼을 클릭하면
Then:
  - DB에 중복 저장되지 않는다
  - "이미 사용 중인 이메일입니다" 인라인 에러 메시지가 표시된다
```

### MOD_B-01. {{핵심 기능 B 생성}}

```
Given: 로그인된 사용자가 유효한 정보를 입력했을 때
When: 등록 버튼을 클릭하면
Then:
  - DB에 PENDING 상태로 저장된다
  - 목록 화면으로 리디렉션된다
  - 새로 생성된 항목이 목록 상단에 표시된다

Given: 동시에 같은 자원에 N명이 접근했을 때
When: 모두 등록 버튼을 클릭하면
Then:
  - 중복 또는 데이터 정합성 오류 없이 처리된다 (동시성 보장)
```

### EXT-01. 외부 API 연동

```
Given: 외부 API가 정상 응답하는 환경에서
When: {{연동 기능}}을 실행하면
Then:
  - 외부 API 응답이 처리되어 결과가 표시된다
  - 응답 시간 P95 ≤ 3초

Given: 외부 API가 타임아웃/오류 응답하는 환경에서
When: {{연동 기능}}을 실행하면
Then:
  - 폴백 결과가 사용자에게 표시된다
  - 에러가 로그에 기록된다
  - 시스템이 중단되지 않는다
```
