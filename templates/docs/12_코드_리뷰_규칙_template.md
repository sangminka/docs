## 1. 기본 원칙 (마인드셋)
- 코드가 아니라 **변경의 의도**를 리뷰한다.
- 모르면 묻는다 — `[question]` 태그로 부담 없이.
- 한 PR에 너무 많은 지적 금지. **블로킹 이슈만** `[must]`.
- 24시간 내 1차 피드백 목표.

## 2. PR 작성 규칙

### 2.1 제목 형식
`{type}({scope}): {한 줄 요약}` + 필요 시 `[WIP]`

예: `feat(reservation): 전화 예약 등록 흐름 추가`

### 2.2 본문 필수 항목

```markdown
## 변경 요약
- ...

## 주요 변경 파일
- `A.java` — ...
- `B.mustache` — ...

## 테스트 방법
1. ...
2. ...

## 특별히 봐줬으면 하는 부분
- ...

## 관련 이슈
- closes #123
```

### 2.3 PR 크기 기준
- ✅ 권장: 400줄 이하
- ⚠️ 주의: 400~1000줄 (이유 설명 필수)
- ❌ 금지: 1000줄 초과 (분할 요청)

## 3. 리뷰 체크리스트

### ✅ A. 보안 & 안전 — **0순위 (가장 먼저)**
- [ ] 인증·인가 누락 없는가 (Role 기반 제약)
- [ ] SQL Injection / XSS 위험 없는가
- [ ] 민감 정보 로그/응답 노출 없는가
- [ ] CSRF 토큰 처리
- [ ] 외부 API 키 하드코딩 없는가

### ✅ B. 외부 연동({{LLM/결제 등}}) 특화 — **1순위**
- [ ] 타임아웃 설정
- [ ] 실패 시 폴백 경로
- [ ] 재시도 정책 적절한가
- [ ] 테스트에서 실제 호출 없음

### ✅ C. 기능·로직 — **2순위**
- [ ] 상태 전이가 ERD 정의와 일치
- [ ] 엣지 케이스 처리 (빈 값, null, 동시성)
- [ ] 비즈니스 규칙 위반 없음

### ✅ D. 코드 품질 & 가독성 — **3순위**
- [ ] 네이밍 명확
- [ ] 책임 분리 (Controller-Service-Repo)
- [ ] 중복 코드 없음
- [ ] 매직 넘버 없음

### ✅ E. 테스트 & 문서 — **4순위**
- [ ] 신규 Service 메서드에 테스트
- [ ] 실패 케이스 포함
- [ ] API 명세서 동기화

## 4. 코멘트 작성 가이드

### 4.1 중요도 태그
| 태그 | 의미 |
| --- | --- |
| `[must]` | 반드시 수정 — 미수정 시 merge 불가 |
| `[should]` | 강하게 권장 — 이유가 있으면 유지 가능 |
| `[nit]` | 취향 / 사소 — 무시 가능 |
| `[question]` | 질문 / 확인 |
| `[praise]` | 칭찬 |

### 4.2 코멘트 예시
```
[must] 이 경로는 @PreAuthorize("hasRole('ADMIN')") 필요합니다.
일반 ROLE_STAFF도 접근 가능해 보여요.

[should] 이 메서드 이름이 실제 동작과 달라 보입니다.
`createIfAbsent`가 더 맞아 보여요.

[nit] 한 줄 공백 제거 권장.
```

### 4.3 논쟁이 길어질 때
- 3왕복 넘어가면 DM이나 통화로 해결 → 결론만 PR에 코멘트로 남김
- 결정이 애매하면 **책임개발자 에스컬레이션**

## 5. 리뷰 담당 & 에스컬레이션

### 5.1 기본 리뷰 담당 (6명 기준)

| PR 영역 | 모듈 소유자 | 1차 리뷰 | 2차 리뷰 |
| --- | --- | --- | --- |
| Auth + {{MODULE_A}} | Dev A | Lead | Dev B |
| {{MODULE_B}} (동시성 포함) | Dev B | Lead | Dev A |
| {{MODULE_C}} (외부연동 포함) | Dev C | Lead | Dev D |
| Admin + Stats + UI | Dev D | Lead | Dev C |
| 공유 레이어 (config·common·security) | Lead 전담 | Lead 필수 | 팀 전체 알림 |
| CI/CD·인프라·테스트 자동화 | DevOps/QA | Lead | Dev A |

> **소유권 원칙:** 모듈 경계를 침범하는 PR은 침범 대상 모듈의 소유자를 **반드시 리뷰어에 추가**해야 한다.

### 5.2 외부 연동 PR은 책임개발자 검토 우선.

### 5.3 W4 마지막 3일 운영 규칙
- `[must]` 외 코멘트는 후속 이슈로 넘김
- 최소 승인 1명 + CI green으로 merge 가능

### 5.4 블로킹 시 에스컬레이션
- 24시간 내 리뷰 없으면 Slack `#review` 채널에 핑
- 48시간 초과 시 책임개발자 직접 리뷰

---

## 6. Merge 조건
- [ ] Review 2인 이상 approve (그 중 책임개발자 1명 권장)
- [ ] CI green
- [ ] `[must]` 코멘트 모두 해결
- [ ] develop 리베이스 후 컨플릭트 없음

**Merge 전략:** Squash merge (기본) / Rebase merge (릴리스 브랜치)

## 7. 팀 문화 & 운영 규칙

### 7.1 주간 PR 회고 (매주 금요일 통합 미팅 중 10분)
- 이번주 merge된 PR 수 / 평균 리뷰 시간
- 가장 큰 PR / 가장 빠르게 merge된 PR 공유

### 7.2 핑(Ping) 규칙
- 리뷰 요청 후 24h 응답 없음 → 1회 DM 핑
- 48h 이상 블로킹 → 책임개발자 에스컬레이션

### 7.3 주차별 리뷰 강도

| 주차 | 강도 |
| --- | --- |
| W1 | 높음 — 구조 / 패턴 잡기 |
| W2~W3 | 보통 |
| W4 | 낮음 — `[must]` 중심 |
