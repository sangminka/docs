## 1. 작성 순서 & 의존 관계

```
프로젝트 계획서
    ├─> ERD                          (스키마 확정)
    │     └─> 아키텍처 정의서        (패키지·소유자·공유 레이어)
    │           ├─> API 명세서        (URL·메서드·반환 규칙)
    │           │     ├─> 시퀀스 다이어그램
    │           │     └─> 화면 기능 정의서
    │           └─> Git 규칙 + 코드리뷰 규칙
    ├─> 요구사항 정의서 (교과 요건 충족 체크)
    ├─> 스토리보드     (→ 화면 기능 정의서와 교차검증)
    ├─> 테스트 전략서  (ERD·API 확정 후 작성)
    └─> 배포 가이드 + 환경 설정 템플릿
```

**실행 포인트**
- **ERD → API → 화면**이 일방향으로 정합되어야 한다. 중간에 스키마가 바뀌면 API와 화면도 반드시 버전 업.
- 문서 버전은 `v1.0`으로 시작해서 **정합 불일치를 고칠 때마다 마이너 버전을 올린다** (HMS는 계획서가 v4.3까지 갔음).
- 매 버전 상단에 "변경 이력" 표를 붙여 diff를 요약.

## 2. 문서별 작성 컨벤션

### 파일 이름
- `{프로젝트이모지}_{프로젝트명}_—_{문서이름}_v{버전}.md`
- 공백은 `_`로, em dash `—` 사용.
- 버전 표기: `v1`, `v2`, `v4_3`처럼 점을 `_`로 치환(파일명 안전).

### 문서 상단 메타데이터 (공통)
```markdown
> **문서 버전:** v1.0
> **작성일:** YYYY-MM-DD
> **개발 기간:** N주
> **팀 구성:** 책임개발자 1명 + 개발자 N명
> **연관 문서:** ERD vX / API 명세서 vX / 화면 정의서 vX
```

### 변경 이력 표
문서 두 번째 섹션에 항상 아래 형태 유지:
```markdown
## 📌 v{이전} → v{현재} 변경 사항

| 항목 | v{이전} | v{현재} |
| --- | --- | --- |
| ... | ... | ... |
```

## 3. 프로젝트 계획서 핵심 섹션

1. **개요** — SMART 목표 / 전체 시스템 구조(ASCII 박스 다이어그램)
2. **특수 기능 상세 정의** — LLM 같은 핵심 기능은 화면 흐름 + API 호출 설계 + 시스템 프롬프트 모두 문서화
3. **추가 테이블 정의** — ERD 업데이트 예고
4. **Scope Definition** — In-Scope 체크리스트 + Out-Scope 명시 (Scope Lock)
5. **WBS & Gantt** — 압축 일정일수록 "W4 병렬 작업 구조" 명시
6. **인력 배치표** — 역할·담당 영역·주차별 분배·성공 조건
7. **커뮤니케이션 계획** — 일일 스탠드업, 주간 통합 미팅, 채널
8. **형상관리 전략** — Git 브랜치 전략 / 커밋 컨벤션 / 버전 로드맵
9. **리스크 관리** — 최우선 리스크 2가지 + 대응 타임라인

## 4. ERD 문서 핵심 섹션

- **1. ERD 전체 다이어그램** — Mermaid `erDiagram`
- **2. 테이블 정의서** — 테이블별로 `컬럼 / 타입 / NULL / 기본값 / 설명 / PK·FK·UK`
- **3. 관계 정의** — 관계 목록 + 외래 키(FK) 목록 분리
- **4. 상태 코드 정의** — 상태 흐름 다이어그램 + 상태별 상세 정의 + Role 코드
- **5. 변경 내역**
- **6. 설계 결정 사항 & 주의사항** — 핵심 설계 결정 N가지 / MVP에서 제외한 설계 / 인덱스 권장 / JPA 엔티티 패키지 구조

> **좋은 ERD 문서의 특징:** "왜 이 스키마를 골랐는가"와 "의도적으로 뺀 것"이 같이 적혀 있다.

## 5. 아키텍처 정의서 핵심 섹션

- **1. 전체 시스템 레이어 구조**
- **2. 기술 스택** (버전까지 명시)
- **3. Spring Boot 패키지 구조 & 소유자 맵핑** — 각 패키지에 담당자를 박는 것이 핵심
- **4. 담당자별 소유 영역 전체 지도** — 팀원 N명 각각 소유 파일 목록
- **5. 공유 레이어 — 책임개발자 단독 소유** (예: 슬롯 관리, 공통 인터셉터, 엔티티 — 변경 금지)
- **6. 기능 모듈별 경계 정의** — 모듈 A/B/C... 로 나누고 각 모듈의 인풋/아웃풋 정의
- **7. 세션 저장소 설정** (Redis 등)
- **8. 서버 사이드 유효성 검증 패턴** — `@Valid` / SSR PRG 패턴 / REST API 패턴 / 에러 표시
- **9. CSRF 토큰 처리** — Mustache 메타 태그 + AJAX 설정
- **10. 페이징 구현 패턴** — `Pageable`, 공통 컨트롤러 패턴

> **핵심 원칙:** "기능 단위 소유권 분리" — Controller·Service·Repository를 동일 담당자가 소유해야 파일 충돌이 안 난다.

## 6. API 명세서 핵심 섹션

### 1. 공통 규칙 (반드시 최상단)
- Base URL
- URL 설계 원칙 (RESTful vs RPC 스타일 선택)
- URL 접두어별 접근 권한 (`/admin/**`, `/doctor/**`, `/api/**`...)
- HTTP 메서드 규칙
- 컨트롤러 반환 규칙 (SSR은 `String`, REST는 `ResponseEntity<T>`)
- JSON API 레이어 (`/api/**`) 분리
- 서버 사이드 유효성 검증 패턴
- 인증 오류 공통 처리

### 2. 엔드포인트 정의 템플릿 (항목마다 반복)
```markdown
### N.N {기능명}

- **Method + URL:** `GET /path/{id}`
- **인증:** ROLE_STAFF
- **Request Params / Body:** ...
- **Response:**
  - 성공: 200 + JSON 스키마
  - 실패: 4xx + 에러 스키마
- **컨트롤러 반환:** Mustache view name 또는 JSON
- **비고:** 페이징·필터·검색 파라미터
```

### 3. 섹션 순서 (역할별 그룹화)
인증 → 외부(비회원) → 특수 기능(LLM) → ROLE_A → ROLE_B → ... → ROLE_ADMIN → 공통(내 정보관리)

## 10. 코드 리뷰 규칙 핵심 섹션

- PR 제목 형식: `[Type] {주제} (#이슈번호)` (Conventional Commits 기반)
- PR 본문 필수 항목: 변경 요약 / 주요 변경 파일 / 테스트 방법 / 특별 리뷰 포인트 / 관련 이슈
- **PR 크기:** 400줄 이하 권장, 1000줄 초과 시 분할 요청
- 리뷰 체크리스트 우선순위:
  1. 보안·안전 (인증/인가, SQL Injection, XSS, 민감 로그)
  2. 외부 연동(LLM 등) 특화 — 키 노출, 타임아웃, 폴백
  3. 기능·로직
  4. 코드 품질 & 가독성
  5. 테스트 & 문서
- 코멘트 태그: `[must]` / `[should]` / `[nit]` / `[question]`
- Merge 조건: 리뷰 2인 이상 approve + CI green + 컨플릭트 해결
- 주차별 리뷰 강도: W4 마지막 3일은 `[must]` 외 코멘트는 후속 이슈로

## 11. Git 규칙 핵심 섹션

- **브랜치 전략:** `main`(배포 전용) / `develop`(개발 중심) / `feature/...` `bugfix/...` `hotfix/...`
- **데일리 루틴:**
  1. `git switch develop && git pull`
  2. `git switch -c feature/123-short-desc`
  3. 작업 → 커밋 → 푸시
  4. PR 생성 → 리뷰 → **Squash merge**
  5. develop 최신화 후 rebase
- **Commit Message 규칙 (Conventional Commits):**
  - `feat:` / `fix:` / `refactor:` / `test:` / `docs:` / `chore:` / `ci:` / `style:`
  - 팀원/팀장 구분 prefix 추가 (예: `[lead]`, `[devA]`)
- **PR 템플릿** — 관련 이슈 / 변경 요약 / 스크린샷 / 리뷰 포인트 / 체크리스트
