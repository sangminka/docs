## 1. 문서 목적

본 문서는 QT-AI 앱에서 사용자가 이용할 수 있어야 하는 기능과 시스템이 충족해야 하는 요구사항을 정의한다. 문서의 목적은 비즈니스 이해관계자, 기획자, 디자이너, 개발자, 검토자가 서비스의 범위와 동작 기준을 동일하게 이해하도록 돕는 것이다.

본 문서는 “무엇을 만들 것인가”와 “시스템이 어떤 조건에서 어떻게 동작해야 하는가”를 중심으로 작성한다. 테이블 구조, 컬럼, 인덱스, API 요청·응답 스키마, 배포 구성, 외부 서비스 연동 세부 방식은 별도 설계 문서에서 다룬다.

요구사항은 기능 ID(F-01 등)를 기준으로 추적한다. 각 기능은 우선순위를 `필수`, `중요`, `권장`으로 구분하며, 구현·운영 리스크는 `고`, `중`, `저`로 표시한다. 리스크 수준은 구현 난이도뿐 아니라 저작권, 개인정보, AI 검증, 데이터 보존, 운영 책임의 복합 위험도를 함께 고려한다.

각 기능은 후속 문서와 연결되어야 한다. 상세 ERD, API 스키마, 인프라 구성, 상세 와이어프레임, 기술 스택, 일정 계획은 본 문서에서 확정하지 않고 별도 문서에서 다룬다. 다만 본 문서는 해당 후속 문서가 반드시 다루어야 할 입력 요구사항을 제공한다.

## 2. 프로젝트 개요

## 2.1 서비스 한 줄 정의

QT-AI는 QT와 성경에 익숙하지 않은 사용자도 오늘의 QT 본문을 쉽게 읽고, 한글·영어 성경 본문과 해설, 용어 풀이, 장면 시뮬레이터, 사실 기반 AI 응답, 개인 묵상 기록을 함께 활용할 수 있도록 돕는 QT 보조 앱이다.

## 2.2 해결하려는 문제

| 대상 사용자 | 현재 문제 | 앱의 해결 방향 |
|---|---|---|
| 신앙 입문자 | 성경을 어디서부터 읽어야 할지 막막함 | 오늘의 QT → 해설 → 시뮬레이터 → 묵상 기록의 단계형 흐름 제공 |
| 시간 부족 직장인 | 긴 주석이나 자료를 찾아볼 시간이 부족함 | 필요한 경우에만 요약·해설·TTS를 선택적으로 제공 |
| 묵상 방법론이 부족한 사용자 | 자기중심 해석으로 흐를 위험이 있음 | 객관적 QT 가이드와 사전 작성된 묵상 질문 제공 |
| 기존 성경 AI 앱 사용자 | 답변 출처와 검증 기준이 불명확하고, AI 환각이나 출처 미표시 위험이 있음 | AI 산출물 검증 체계, 출처 표시, 가치 판단 차단 기준 제공 |
| 본문 장면 상상이 어려운 사용자 | 텍스트만으로 시대 배경이나 사건 흐름을 떠올리기 어려움 | QT 본문에 한해 장면 시뮬레이터 제공 |
| 모르는 단어·시대상이 많은 사용자 | 검색 결과의 신뢰도를 판단하기 어려움 | 단어·시대상·역사에 한정한 사실 기반 AI 응답 제공 |
| 자유 기록을 원하는 사용자 | QT 외의 설교, 기도제목, 회개, 감사 기록을 한곳에 남기기 어려움 | 자유 노트 기능과 카테고리별 기록 제공 |
| 꾸준히 묵상하고 싶은 사용자 | 묵상 이력을 한눈에 보기 어려움 | 마이페이지에서 묵상 달력, 통계, 미션 진행률 제공 |

## 2.3 서비스 방향성

QT-AI는 초심자가 쉽게 사용할 수 있는 심플한 개인 중심 QT 보조 앱을 지향한다. 커뮤니티 기능은 보조 기능으로 제공하며, 기본 경험은 개인 묵상과 본문 이해에 집중한다.

AI는 두 가지 방식으로만 동작한다. 첫째, 해설과 시뮬레이터처럼 사전에 생성·검증된 콘텐츠를 사용자에게 조회 형태로 제공한다. 둘째, 사용자가 본문을 읽다가 모르는 단어, 시대상, 역사적 사실을 질문할 경우 단발성 사실 응답을 제공한다. AI는 사용자의 묵상 노트 작성 과정에 개입하지 않으며, 다중 턴 대화, 가치 판단, 신앙 상태 평가, 자율적 묵상 권유를 수행하지 않는다. 모든 AI 산출물은 생성 지시 자산 버전, 검증 체크리스트 버전, 산출물 생성 시각, 검증 상태를 추적할 수 있어야 한다.

모든 사용자의 기록은 기본적으로 비공개이며, 사용자가 명시적으로 공유를 선택한 경우에만 닉네임이 포함되어 공유된다. 공유 화면에는 닉네임이 함께 공개된다는 안내를 표시해야 한다.

## 3. 핵심 사용자 흐름

### 3.1 1차 시연 우선 흐름

6명/1개월 프로젝트에서 시연 가능한 완성도를 확보하기 위해 1차 개발은 다음 흐름을 우선 완성한다.

1. 사용자가 카카오 로그인으로 앱에 진입한다. (F-04)
2. 오늘의 QT 본문을 한글·영어 절 단위로 조회한다. (F-01)
3. 필요한 경우 AI 요약, 해설, 용어 풀이를 확인한다. (F-08)
4. 오늘의 QT 화면에서 QT 노트를 작성하고 수동 저장한다. (F-03)
5. 마이페이지에서 묵상 달력과 기록 상태를 확인한다. (F-13)

이 흐름은 기능 제외를 의미하지 않는다. 기능 ID는 유지하며, 이후 AI 콘텐츠 사전 생성·검증 관리, 관리자 기능, 커뮤니티, TTS, 시뮬레이터, 찬양 기능을 순차적으로 붙인다.

### 3.2 전체 사용자 흐름

사용자가 앱을 처음 실행하면 회원가입·로그인 화면으로 진입한다. 앱의 모든 기능은 로그인 후에만 사용할 수 있으며, 비회원 접근은 허용하지 않는다. 사용자는 카카오 로그인을 통해 가입·로그인할 수 있고, 한 번 로그인한 이후에는 유효한 인증 상태가 유지되는 동안 자동 로그인이 적용된다.

로그인 직후 첫 실행 사용자는 QT가 무엇인지, 앱을 어떤 순서로 사용하면 되는지 안내하는 튜토리얼을 확인한다. 튜토리얼 종료 후 사용자는 오늘의 QT 화면으로 이동한다.

오늘의 QT 화면에서 사용자는 절 단위 한글·영어 성경 본문을 읽고, 필요한 경우 한 줄 요약, 용어 풀이, 절별 해설을 펼쳐서 확인한다. 본문 옆 시뮬레이터 아이콘을 누르면 QT 본문의 장면을 시각화한 짧은 애니메이션을 볼 수 있다. 본문이나 해설은 TTS를 통해 음성으로 들을 수 있다.

사용자가 모르는 단어, 시대상, 역사적 사실을 질문하고 싶을 경우 우측 하단 AI 버튼을 누른다. AI 응답은 단발성으로 제공되며, 이전 질문 맥락을 유지하지 않는다. 질문이 가치 판단, 신앙 평가, 주관적 조언, 단정적 설교 요청에 해당하는 경우 시스템은 답변을 생성하지 않고 제한 안내를 표시한다. 각 질문은 독립 세션으로 처리되며, 사용자가 직전에 입력한 질문이나 이전 응답을 전제로 이어서 답변하지 않는다.

사용자는 오늘의 QT에 연결된 QT 노트를 작성할 수 있다. QT 노트는 느낀 점, 기억할 구절, 적용할 점, 기도의 4개 섹션으로 구성된다. 사용자가 “임시저장” 버튼을 누르면 작성 중 상태가 보관되고, “저장” 버튼을 누르면 확정본으로 저장된다. 자동 저장은 어느 단계에서도 동작하지 않는다.

사용자는 오늘의 QT 외에도 자유 노트를 작성할 수 있다. 자유 노트는 설교 노트, 기도제목 노트, 회개 노트, 감사일기로 구분된다. 자유 노트는 QT 노트와 달리 일자별 1건 제한이 적용되지 않는다.

사용자는 묵상 기록을 기본 비공개 상태로 보관한다. 사용자가 공유를 선택하면 해당 기록의 공유본이 생성되고, 공유본에는 닉네임이 노출된다. 공유본은 원본 기록이 이후 수정되더라도 공유 당시의 본문을 유지하는 변경 불가 스냅샷으로 관리한다.

회원 탈퇴 후 재가입하는 사용자는 이전 계정의 비공개 기록, 공유 기록, 알림, AI Q&A 기록을 복구할 수 없다는 안내를 확인해야 한다. 법적 보존 대상 데이터가 존재하더라도 해당 데이터는 복구 목적이 아니라 법적·운영적 보존 목적에 한해 제한적으로 관리된다.

## 4. 용어 정의

| 용어 | 정의 |
|---|---|
| QT | 개인이 일정 시간 성경을 읽고 묵상·적용·기도를 정리하는 신앙 활동 |
| 오늘의 QT | 매일 정해지는 하루치 묵상용 성경 본문. 장·절 중간 끊김을 허용함 |
| QT 노트 | 오늘의 QT 본문에 연결된 1일 1건의 노트. 느낀 점, 기억할 구절, 적용할 점, 기도의 4개 섹션으로 구성됨 |
| 자유 노트 | 사용자가 QT 본문과 무관하게 작성하는 노트. 설교 노트, 기도제목 노트, 회개 노트, 감사일기를 포함함 |
| 묵상 기록 | QT 노트와 자유 노트를 포괄하는 상위 개념 |
| 설교 노트 | 자유 성경 본문을 선택하여 작성하는 노트 |
| 기도제목 노트 | 본문 좌표와 무관하게 기도제목을 정리하는 노트 |
| 회개 노트 | 본문 좌표와 무관하게 개인 회개 내용을 기록하는 노트 |
| 감사일기 | 본문 좌표와 무관하게 일상 감사 내용을 기록하는 노트 |
| 해설 | 사용자가 본문을 이해할 수 있도록 풀어 쓴 설명. AI 생성 및 주석 기반 콘텐츠를 포함함 |
| 용어 풀이 | 본문 이해에 필요한 단어, 시대상, 배경지식에 대한 짧은 설명 |
| 시뮬레이터 | 오늘의 QT 본문 장면을 보여주는 짧은 애니메이션 클립 |
| 장면 스크립트 | 시뮬레이터 생성을 위해 사용되는 구조화 데이터 |
| 컴포넌트 라이브러리 | 시뮬레이터 장면을 구성하기 위해 시스템이 보유하는 배경, 캐릭터, 전환, 자막 등의 자산 목록 |
| 찬양 큐레이션 | 운영자가 사전에 등록·관리하는 추천 찬양 곡 목록 |
| 닉네임 나눔 | 사용자가 선택한 묵상 기록을 닉네임 노출 상태로 공유하는 공간 |
| 사실 기반 Q&A AI | 단어·시대상·역사에 한정하여 단발성으로 응답하는 AI 기능 |
| 다층 검증 | AI 산출물을 사용자에게 노출하기 전에 형식, 의미, 인간 검토 기준으로 점검하는 품질 관리 체계 |
| 검증 체크리스트 | AI 산출물이 신학 기준, 사실 정합성, 금지 표현 기준을 벗어나지 않는지 점검하는 항목 묶음 |
| 신학 기준 문서 | 서비스가 정통으로 인정하는 신앙 기준과 출처 묶음 |
| 공유본 스냅샷 | 사용자가 공유를 선택한 시점의 본문, 노트 내용, 닉네임, 공개 설정을 별도 보존한 복제본. 원본 수정·삭제의 영향을 받지 않음 |

| 임시본 | 사용자가 임시저장한 작성 중 상태의 QT 노트 데이터 |
| 확정본 | 사용자가 저장 버튼을 눌러 최종 저장한 QT 노트 데이터 |
| 임시저장 | 사용자가 “임시저장” 버튼을 명시적으로 눌러 작성 중 상태를 보관하는 동작 |
| 저장 | 사용자가 “저장” 버튼을 눌러 노트를 확정본으로 저장하는 동작 |

## 6.2 F-01 — 성경·QT 본문 화면

### 기능 설명

사용자는 로그인 후 오늘의 QT 본문을 절 단위로 읽을 수 있어야 한다. 본문은 한글과 영어를 비교할 수 있는 형태로 제공한다. 앱 진입 후 별도 홈 화면 없이 오늘의 QT 화면으로 이동한다.

일반 성경 조회 화면에서는 책, 장, 절을 자유롭게 선택할 수 있다. 단, 해설과 시뮬레이터는 오늘의 QT 범위와 연결된 콘텐츠를 우선으로 제공한다.

### 요구사항

- 로그인하지 않은 사용자는 성경·QT 본문 화면에 접근할 수 없어야 한다.
- 오늘의 QT는 날짜별 책·장·절 범위 기준으로 제공되어야 한다.
- QT 범위는 장·절 중간에서 시작하거나 끝날 수 있어야 한다.
- 한글 성경 본문과 영어 성경 본문은 절 좌표를 기준으로 정렬되어야 한다.
- 해설, 시뮬레이터, AI 질문, TTS 기능은 본문 화면에서 접근 가능해야 한다.
- 해설이나 시뮬레이터 로딩이 실패해도 본문은 가능한 경우 우선 표시되어야 한다.

### 데이터 정책

QT 본문 큐레이션 자료에는 본문 텍스트를 저장하지 않고 날짜별 범위 정보만 저장한다. 실제 본문 텍스트는 저작권 문제가 없는 성경 데이터셋에서 범위 정보를 기준으로 조회한다.

### 수용 기준

- 사용자가 로그인 상태로 앱을 실행하면 오늘의 QT 화면으로 진입해야 한다.
- 오늘의 QT 본문은 한글·영어가 동일 절 기준으로 표시되어야 한다.
- 해설 생성 여부와 무관하게 성경 본문은 표시 가능해야 한다.
- 로그인하지 않은 사용자는 해당 화면에 접근할 수 없어야 한다.

### 예외 처리

- 오늘의 QT 범위 데이터가 없으면 “오늘의 QT가 아직 등록되지 않았습니다” 안내를 표시한다.
- 성경 본문 조회에 실패하면 재시도 버튼과 오류 안내를 표시한다.
- 영어 본문만 실패한 경우 한글 본문을 우선 표시하고 영어 본문 영역에는 오류 상태를 표시한다.

## 6.3 F-02 — AI 콘텐츠 사전 생성

### 기능 설명

시스템은 사용자에게 노출할 해설과 시뮬레이터 콘텐츠를 사전에 생성한다. 사용자 요청 시점에는 실시간 생성이 아니라 검증을 통과한 사전 생성 콘텐츠를 조회한다.

### 요구사항

- 해설은 본문과 입력용 주석 자료를 기반으로 생성되어야 한다.
- 시뮬레이터는 검증된 해설과 장면 스크립트를 기반으로 생성되어야 한다.
- 생성 결과는 검토 대기, 노출 가능, 반려 상태로 관리되어야 한다.
- 검증을 통과하지 못한 콘텐츠는 사용자에게 노출되어서는 안 된다.
- 모든 사용자 노출 콘텐츠에는 출처 표기 정책이 적용되어야 한다.
- AI 산출물은 생성 시 사용된 지시 자산 버전과 검증 체크리스트 버전을 추적할 수 있어야 한다.

### 수용 기준

- 배치 생성 결과는 상태값을 가져야 한다.
- 노출 가능 상태가 아닌 해설은 사용자 화면에 표시되지 않아야 한다.
- 반려된 산출물은 반려 사유가 로그로 남아야 한다.
- 동일 본문에 대해 재생성한 경우 이전 산출물과 신규 산출물을 구분할 수 있어야 한다.

### 예외 처리

- AI 생성 실패 시 해당 본문 콘텐츠는 검토 대기 또는 생성 실패 상태로 기록한다.
- 생성 실패가 반복될 경우 관리자 화면에서 확인 가능해야 한다.
- 해설이 없는 절은 사용자 화면에 “준비된 해설 없음”으로 표시한다.

## 6.4 F-03 — QT 노트 작성·관리

### 기능 설명

사용자는 오늘의 QT 화면에서 QT 노트를 작성할 수 있다. QT 노트는 느낀 점, 기억할 구절, 적용할 점, 기도의 4개 섹션으로 구성된다. 일부 섹션만 작성해도 저장할 수 있다.

### 저장 정책

| 동작 | 트리거 | 결과 표시 |
|---|---|---|
| 임시저장 | 사용자가 “임시저장” 버튼 클릭 | “임시저장되었습니다” 표시 |
| 저장 | 사용자가 “저장” 버튼 클릭 | “저장 완료되었습니다” 표시 |

자동 저장은 어떤 단계에서도 동작하지 않는다. 사용자가 명시적으로 버튼을 누른 경우에만 데이터가 저장된다.

### 요구사항

- 작성 화면에는 “임시저장”과 “저장” 버튼이 제공되어야 한다.
- QT 노트는 사용자별·날짜별로 1건만 생성되어야 한다.
- 기존 QT 노트를 다시 열 경우 현재 저장 상태를 확인할 수 있어야 한다.
- QT 노트는 기본 비공개 상태로 생성되어야 한다.
- AI는 QT 노트 내용을 생성, 수정, 평가, 추천하지 않아야 한다.
- QT 노트 작성·수정·삭제 이벤트는 묵상 달력과 연동되어야 한다.

### 수정·삭제 정책

- 사용자는 본인이 작성한 QT 노트를 수정할 수 있다.
- 사용자는 본인이 작성한 QT 노트를 삭제할 수 있다.
- QT 노트가 삭제되면 해당 날짜의 묵상 완료 표시도 갱신되어야 한다.
- 공유된 기록이 있는 경우 원본 QT 노트를 수정하거나 삭제해도 공유본은 공유 당시 본문을 유지한다.

### 수용 기준

- 사용자가 저장 버튼을 누르기 전에는 확정본이 생성되지 않아야 한다.
- 사용자가 임시저장 버튼을 누르면 임시본이 저장되어야 한다.
- 사용자가 저장 버튼을 누르면 확정본이 저장되어야 한다.
- 같은 사용자가 같은 날짜에 QT 노트를 중복 생성할 수 없어야 한다.
- 자동 저장으로 인해 사용자가 삭제한 내용이 다시 저장되는 상황이 발생해서는 안 된다.

### 예외 처리

- 저장 실패 시 사용자는 실패 안내와 재시도 버튼을 확인할 수 있어야 한다.
- 네트워크 오류로 저장에 실패한 경우 저장되지 않았음을 명확히 표시해야 한다.
- 다른 기기에서 동일 날짜 노트가 수정된 경우 자동 병합하지 않고 충돌 안내를 제공해야 한다. 병합 방식 또는 최종 저장본 선택 방식은 후속 데이터 정책에서 확정한다.

## 6.5 F-04 — 회원 인증

### 기능 설명

사용자는 카카오 로그인을 통해 가입·로그인한다. 앱의 모든 기능은 로그인 후에만 사용할 수 있다. 유효한 인증 상태가 유지되는 동안 자동 로그인이 적용된다.

### 요구사항

- 카카오 로그인을 1차 로그인 수단으로 제공한다.
- 앱 설치 후 첫 진입 시 로그인하지 않은 사용자는 로그인 화면으로 이동해야 한다.
- 로그인하지 않은 사용자는 어떤 기능도 사용할 수 없어야 한다.
- 자동 로그인은 토큰 유효성 검증을 통해 유지되어야 한다.
- 사용자는 회원 탈퇴를 신청할 수 있어야 한다.
- 탈퇴 신청 시 본인 확인 절차를 거쳐야 한다.

### 회원 탈퇴 정책

회원 탈퇴가 완료되면 계정은 즉시 비활성화된다. 법적 보존 의무가 있는 데이터는 정해진 기간 동안 제한된 권한으로 보관하고, 보존 기간이 종료되면 영구 삭제한다. 정확한 보존 기간과 보존 대상은 별도 회원 탈퇴 보존 정책 문서에서 확정한다.

탈퇴 시 처리 대상은 다음과 같이 구분한다.

| 데이터 | 처리 원칙 |
|---|---|
| 비공개 QT 노트 | 법적 보존 대상이 아니면 삭제 또는 비식별화 |
| 비공개 자유 노트 | 법적 보존 대상이 아니면 삭제 또는 비식별화 |
| 공유글 | 서비스 정책에 따라 삭제, 비공개, 또는 작성자 비식별화 처리 |
| 댓글 | 정책에 따라 삭제 또는 작성자 비식별화 처리 |
| 좋아요 | 사용자 식별 정보 제거 |
| 신고 기록 | 운영상 필요한 기간 동안 제한 보관 가능 |
| AI Q&A 기록 | 저장 여부와 보존 기간을 별도 정책으로 명시 |

### 수용 기준

- 로그인하지 않은 사용자는 앱 기능 화면에 접근할 수 없어야 한다.
- 카카오 로그인 성공 후 사용자는 오늘의 QT 화면 또는 첫 실행 튜토리얼로 이동해야 한다.
- 앱 재실행 시 유효한 토큰이 있으면 자동 로그인되어야 한다.
- 회원 탈퇴 후 계정은 즉시 비활성 상태가 되어야 한다.
- 탈퇴 후 재가입 시 이전 데이터 복구가 불가하다는 안내가 표시되어야 한다.

### 예외 처리

- 카카오 로그인 실패 시 실패 사유와 재시도 안내를 표시한다.
- 토큰이 만료되었거나 유효하지 않으면 로그인 화면으로 이동한다.
- 탈퇴 처리 중 오류가 발생하면 사용자에게 실패 안내를 표시하고 계정 상태를 변경하지 않는다.

## 6.6 F-05 — 인앱 알림

### 기능 설명

사용자는 마이페이지의 알림 위젯 또는 알림 목록에서 자신의 활동 결과와 시스템 공지를 확인할 수 있다. MVP 기준으로 실시간 푸시 알림은 제공하지 않고, 앱 내 로그형 알림을 제공한다.

### 알림 유형

| 유형 | 생성 조건 |
|---|---|
| 좋아요 알림 | 내가 공유한 글에 다른 사용자가 좋아요를 누른 경우 |
| 댓글 알림 | 내가 공유한 글에 댓글이 작성된 경우. 단, 해당 게시글의 댓글이 ON 상태여야 함 |
| 신고 처리 결과 | 내가 접수한 신고가 처리된 경우 |
| 시스템 공지 | 관리자가 사용자 공지를 발행한 경우 |

### 요구사항

- 알림은 읽음·미읽음 상태를 가져야 한다.
- 사용자는 알림 목록에서 알림 발생 시각과 내용을 확인할 수 있어야 한다.
- 동일 이벤트에 대한 중복 알림은 방지되어야 한다.
- 알림은 마이페이지 위젯에서 요약 형태로 표시되어야 한다.

### 수용 기준

- 내 공유글에 좋아요가 발생하면 알림이 생성되어야 한다.
- 내 공유글에 댓글이 작성되면 알림이 생성되어야 한다.
- 신고 처리 결과가 확정되면 신고자에게 알림이 생성되어야 한다.
- 사용자가 알림을 확인하면 읽음 상태로 변경되어야 한다.

### 예외 처리

- 알림 목록 조회 실패 시 재시도 안내를 표시한다.
- 삭제되거나 비공개 처리된 게시글과 연결된 알림은 상세 이동이 제한될 수 있다.

## 6.7 F-06 — 관리자 운영 기능

### 기능 설명

관리자는 서비스 운영에 필요한 콘텐츠와 사용자 활동을 관리한다. 관리자 기능은 일반 사용자 화면과 분리되어야 하며, 관리자 작업은 감사 로그로 기록되어야 한다.

### 요구사항

- 관리자만 접근 가능한 별도 화면 또는 접근 경로가 있어야 한다.
- 관리자 권한이 없는 사용자는 관리자 기능에 접근할 수 없어야 한다.
- 관리자는 오늘의 QT 범위를 등록·수정할 수 있어야 한다.
- 관리자는 사용자 노출 해설의 상태를 확인하고 승인·반려할 수 있어야 한다.
- 관리자는 신고된 게시글과 댓글을 확인하고 처리할 수 있어야 한다.
- 관리자는 찬양 큐레이션 항목을 등록·수정·삭제할 수 있어야 한다.
- 관리자 작업은 감사 로그로 기록되어야 한다.
- AI Q&A 차단 사례, 자동 검증 실패, 평가 셋 반영, 검증 체크리스트 버전 변경, 컴포넌트 라이브러리 버전 변경도 감사 로그 대상에 포함되어야 한다.

### 감사 로그 필수 대상

- 오늘의 QT 범위 등록·수정·삭제
- 사용자 노출 해설 상태 변경
- AI 재생성 트리거
- 찬양 큐레이션 변경
- 닉네임 나눔 게시글 삭제·비공개 처리
- 댓글 삭제·비공개 처리
- 사용자 제재 또는 권한 변경
- AI Q&A 차단 사례
- 자동 검증 실패 및 반려 사유
- 평가 셋 반영 이력
- 검증 체크리스트 버전 변경
- 컴포넌트 라이브러리 버전 변경

### 권한 제한

관리자는 검증용 한국어 주석 원문을 조회할 수 없다. 검증용 자료는 자체 제작자 역할과 시스템 배치 경로에서만 접근 가능해야 한다.

### 수용 기준

- 일반 사용자가 관리자 URL 또는 화면에 접근하면 접근 거부되어야 한다.
- 관리자가 해설을 반려하면 사용자 화면에는 해당 해설이 노출되지 않아야 한다.
- 관리자의 주요 변경 작업은 작업자, 시각, 대상, 변경 내용과 함께 기록되어야 한다.

### 예외 처리

- 관리자 작업 저장 실패 시 상태 변경이 적용되지 않아야 한다.
- 신고 처리 중 오류가 발생하면 처리 상태를 변경하지 않고 오류 로그를 남긴다.

## 6.8 F-07 — 첫 실행 튜토리얼·QT 가이드

### 기능 설명

첫 실행 사용자는 로그인 직후 QT의 의미와 앱 사용 순서를 안내받는다. 튜토리얼은 초심자가 이해할 수 있는 표현으로 작성한다.

### 요구사항

- 첫 실행 사용자는 튜토리얼을 확인해야 한다.
- 튜토리얼 종료 후 오늘의 QT 화면으로 이동해야 한다.
- 사용자는 튜토리얼 내용을 다시 확인할 수 있어야 한다.
- 앱 내 주요 용어에는 용어 풀이가 제공되어야 한다.

### 수용 기준

- 신규 사용자가 로그인하면 튜토리얼 화면이 표시되어야 한다.
- 튜토리얼 완료 상태가 저장되어야 한다.
- 튜토리얼 완료 사용자는 다음 실행부터 오늘의 QT 화면으로 진입해야 한다.

### 예외 처리

- 튜토리얼 콘텐츠를 불러오지 못하면 기본 안내 문구와 재시도 버튼을 표시한다.
- 튜토리얼 완료 상태 저장에 실패하면 사용자에게 저장 실패를 안내하고 다음 진입 시 튜토리얼을 다시 표시할 수 있다.

## 6.9 F-08 — 본문 요약·해설·용어 풀이

### 기능 설명

사용자는 오늘의 QT 본문에서 필요한 경우 한 줄 요약, 용어 풀이, 절별 해설을 펼쳐서 확인할 수 있다. 해당 콘텐츠는 강제 노출하지 않고 사용자 선택에 따라 표시한다.

### UI 요구사항

| 요소 | 위치 및 동작 |
|---|---|
| 해설 영역 | 해당 큰 대목 바로 아래에 표시 |
| 용어 풀이 | 해설 영역 아래 또는 본문 내 별표 표시와 연결하여 표시 |
| 별표 표시 | 용어 풀이가 있는 단어 옆에 ★ 표시 |
| AI 질문 버튼 | 본문을 가리지 않는 크기로 우측 하단에 표시 |
| AI 사이드바 | 버튼 클릭 시 우측에서 슬라이드 인 또는 본문 옆에 표시 |

### 요구사항

- 사용자가 펼치기 전에는 해설을 강제 노출하지 않아야 한다.
- 해설이 없거나 검증을 통과하지 못한 절은 “준비된 해설 없음”으로 표시해야 한다.
- 모든 해설과 용어 풀이에는 링크 또는 텍스트 형태의 출처·근거가 항상 표시되어야 한다.
- 사용자 화면에서는 실시간 AI 해설 생성을 수행하지 않는다.

### 수용 기준

- 사용자가 해설 펼치기를 선택하면 검증 통과 해설이 표시되어야 한다.
- 검증 통과 해설이 없으면 “준비된 해설 없음”이 표시되어야 한다.
- 용어 풀이가 있는 단어에는 일관된 표시가 적용되어야 한다.
- 해설과 용어 풀이에는 출처 또는 근거가 함께 표시되어야 한다.

### 예외 처리

- 해설 또는 용어 풀이 조회에 실패해도 성경 본문은 유지하고 해당 영역에 재시도 안내를 표시한다.
- 출처 정보가 누락된 콘텐츠는 사용자에게 노출하지 않고 “준비된 해설 없음”으로 처리한다.

## 6.10 F-09 — 찬양 큐레이션·내 찬양 목록

### 기능 설명

사용자는 운영자가 사전 등록한 찬양 곡 목록을 둘러보고, 마음에 드는 곡을 본인 찬양 목록에 저장할 수 있다. AI는 찬양을 추천하지 않는다.

### 요구사항

- 운영자는 찬양 큐레이션 곡을 등록·수정·삭제할 수 있어야 한다.
- 사용자는 찬양 곡을 내 찬양 목록에 저장하거나 저장 해제할 수 있어야 한다.
- 서버에는 곡 메타데이터만 저장한다.
- 가사와 음원 파일은 서버에 저장하지 않는다.
- 사용자 디바이스 음원은 로컬 접근 방식으로만 처리하고 서버에 업로드하지 않는다.

### 데이터 정책

| 데이터 | 저장 여부 |
|---|---|
| 곡명 | 서버 저장 가능 |
| 아티스트명 | 서버 저장 가능 |
| 외부 링크 식별자 | 서버 저장 가능 |
| 가사 | 서버 저장 금지 |
| 음원 파일 | 서버 저장 금지 |
| 사용자 디바이스 음원 파일 | 서버 업로드 금지 |

### 수용 기준

- 사용자는 곡을 내 찬양 목록에 추가할 수 있어야 한다.
- 사용자는 저장한 곡을 목록에서 제거할 수 있어야 한다.
- 서버에는 가사와 음원 파일이 저장되지 않아야 한다.

### 예외 처리

- 디바이스 음원 접근 권한이 거부되면 권한 안내를 표시한다.
- 저장된 외부 링크가 유효하지 않으면 재생 불가 안내를 표시한다.

## 6.11 F-10 — 닉네임 나눔 공간

### 기능 설명

사용자는 본인이 작성한 묵상 기록 중 선택한 기록만 닉네임이 노출된 상태로 공유할 수 있다. 닉네임 나눔 공간은 조용한 나눔을 지향하며, 좋아요, 댓글, 신고의 최소 반응 기능을 제공한다.

### 닉네임 정책

- 사용자는 가입 시 닉네임을 지정해야 한다.
- 닉네임은 중복될 수 없다.
- 공유글에는 실명, 이메일, 전화번호 등 직접 식별정보를 노출하지 않는다.
- 공유글에는 닉네임만 노출한다.
- 닉네임 변경 정책은 별도 정책으로 확정한다. 확정 전까지는 변경 불가로 처리한다.

### 공유 정책

- 모든 묵상 기록은 기본 비공개다.
- 사용자가 공유를 선택한 경우에만 공유본이 생성된다.
- 공유 직전 화면에는 닉네임이 함께 공개된다는 안내를 표시해야 한다.
- 공유본은 공유 당시의 본문을 보존한다.
- 원본 기록이 수정되거나 삭제되어도 공유본은 자동 변경되지 않는다.
- 사용자는 본인이 공유한 글을 비공개 처리하거나 삭제할 수 있어야 한다.

### 댓글 정책

- 작성자는 게시글별로 댓글 ON/OFF를 선택할 수 있다.
- 댓글이 OFF인 게시글에는 새 댓글을 작성할 수 없다.
- 사용자는 본인이 작성한 댓글을 삭제할 수 있어야 한다.
- 관리자는 신고 또는 운영 기준 위반에 따라 댓글을 비공개 또는 삭제 처리할 수 있어야 한다.

### 신고 정책

- 사용자는 공유글과 댓글을 신고할 수 있다.
- 신고된 항목은 관리자 화면에서 확인할 수 있어야 한다.
- 관리자는 신고를 검토하여 유지, 비공개, 삭제, 사용자 제재 중 하나로 처리할 수 있어야 한다.
- 신고 처리 결과는 신고자에게 인앱 알림으로 제공되어야 한다.

### 수용 기준

- 사용자가 공유하지 않은 기록은 다른 사용자에게 보이지 않아야 한다.
- 사용자가 공유를 선택하면 닉네임 노출 안내가 표시되어야 한다.
- 닉네임 중복 가입은 허용되지 않아야 한다.
- 원본 노트를 수정해도 기존 공유본의 본문은 변경되지 않아야 한다.
- 댓글 OFF 게시글에는 댓글 입력창이 비활성화되어야 한다.

### 예외 처리

- 공유 처리 실패 시 공유본이 생성되지 않아야 한다.
- 삭제된 공유글에 접근하면 “삭제되었거나 볼 수 없는 글입니다” 안내를 표시한다.
- 신고가 이미 접수된 동일 항목에 대해서는 중복 신고 안내를 표시할 수 있다.

## 6.12 F-12 — 시뮬레이터 보기

### 기능 설명

사용자는 오늘의 QT 본문 화면에서 시뮬레이터 아이콘을 눌러 본문 장면을 동화책 형태의 짧은 애니메이션으로 볼 수 있다. 시뮬레이터는 오늘의 QT 본문에만 제공한다.

### 적용 범위

| 화면 | 적용 여부 |
|---|---|
| 오늘의 QT 화면 | 적용 |
| 자유 노트 화면 | 미적용 |
| 일반 성경 자유 조회 화면 | 미적용 |

### 요구사항

- 시뮬레이터는 사전 정의된 장면 스크립트와 컴포넌트 라이브러리를 기반으로 생성되어야 한다.
- AI가 자유 형식 코드, 외부 자산, 검증되지 않은 이미지를 직접 생성해 사용자에게 노출해서는 안 된다.
- 시뮬레이터 생성은 해당 본문의 해설이 먼저 검증을 통과한 경우에만 가능해야 한다.
- 인물 외형, 인종, 피부색, 신학적 해석을 단정적으로 표현하지 않아야 한다.
- 시뮬레이터에 사용되는 컴포넌트 라이브러리는 버전으로 관리해야 한다.
- 시뮬레이터 표현은 신학적 중립성 가이드라인을 따라야 하며, 추정에 기반한 장면은 단정적으로 표현하지 않아야 한다.

### 수용 기준

- 오늘의 QT 화면에서 시뮬레이터 아이콘을 누르면 해당 본문의 시뮬레이터가 실행되어야 한다.
- 자유 노트와 일반 성경 조회 화면에서는 시뮬레이터 기능이 제공되지 않아야 한다.
- 검증되지 않은 장면 스크립트는 사용자에게 노출되지 않아야 한다.

### 예외 처리

- 시뮬레이터가 없는 본문에서는 “시뮬레이터가 준비되지 않았습니다” 안내를 표시한다.
- 재생 실패 시 재시도 버튼을 제공한다.

## 6.13 F-13 — 마이페이지

### 기능 설명

사용자는 마이페이지에서 자신의 묵상 이력, 통계, 미션 진행률, 알림, 내 찬양 목록을 확인할 수 있다.

### 구성 위젯

| 위젯 | 내용 |
|---|---|
| 묵상 달력 | QT 완료 날짜 표시, 날짜 클릭 시 해당 날짜 QT 노트로 이동 |
| 월별/주별 통계 | 기도 횟수, 묵상 작성 수 등 단순 카운트 제공 |
| 미션 진행률 | 주간 또는 월간 묵상 목표 달성 상태 표시 |
| 알림 위젯 | F-05의 로그형 알림 요약 표시 |
| 내 찬양 목록 | 사용자가 저장한 찬양 곡 목록 표시 |

### 요구사항

- 기록이 없는 날짜를 클릭해도 빈 기록이 자동 생성되어서는 안 된다.
- QT 노트 생성·수정·삭제 상태는 묵상 달력에 반영되어야 한다.
- 통계는 사용자별로 계산되어야 한다.
- 미션 규칙은 운영 가능한 단순한 기준으로 정의되어야 한다.

### 수용 기준

- QT 노트를 저장한 날짜는 달력에 완료 표시가 되어야 한다.
- QT 노트를 삭제하면 해당 날짜의 완료 표시가 제거되어야 한다.
- 사용자는 월별·주별 작성 수를 확인할 수 있어야 한다.
- 알림 위젯에서 최근 알림을 확인할 수 있어야 한다.

### 예외 처리

- 통계 집계에 실패하면 달력과 기본 사용자 정보는 표시하고 통계 영역에 오류 안내를 표시한다.
- 알림 위젯 조회에 실패하면 마이페이지 전체 진입을 막지 않고 알림 영역에 재시도 안내를 표시한다.

## 6.14 F-14 — AI 산출물 품질 관리

### 기능 설명

AI 산출물의 품질, 사실 정합성, 신학적 안전성, 말투, 출처 표시를 관리하기 위한 검증 체계를 제공한다. 검증 대상은 해설, 시뮬레이터 장면 스크립트, 사실 기반 Q&A 응답을 포함한다.

### 검증 기준

- 신학 기준 문서에 위배되지 않아야 한다.
- 금지 표현과 금지 주제를 포함하지 않아야 한다.
- 성경 본문 좌표와 내용이 일치해야 한다.
- 사용자 신앙 상태를 평가하거나 단정하지 않아야 한다.
- 특정 교단, 인물, 집단을 정죄하지 않아야 한다.
- 출처 없는 단정적 설명을 제공하지 않아야 한다.
- 초심자가 이해할 수 있는 차분한 정보 전달형 말투를 유지해야 한다.

### 다층 검증 구조

| 단계 | 대상 | 검증 주체 | 실패 시 처리 |
|---|---|---|---|
| 1층 형식·규칙 검증 | 스키마, 본문 좌표, 금지 키워드, 자산 카탈로그 위반 여부 | 시스템 | 즉시 반려 |
| 2층 의미·정합성 검증 | 사실 정합성, 금지 주제, 말투, 초심자 이해 가능성 | 시스템 | 반려 또는 검토 대기 |
| 3층 인간 샘플 검토 | 자동 검증이 놓친 뉘앙스, 문화적 적절성, 신학 자문 사항 | 운영진·신학 자문 | 기준 개선 및 재반영 |

### F-15 실시간 응답 검증 방식

사실 기반 Q&A AI는 실시간 응답 특성을 고려하여 1층 형식 검증과 2층 자동 정책 검증을 응답 생성 직후 인라인으로 수행한다. 3층 인간 검토는 모든 응답에 대해 사전 수행하지 않고, 사용자 신고, 샘플링, 운영자 점검을 통한 사후 검토 방식으로 운영한다.

### 요구사항

- 모든 AI 산출물은 생성 지시 자산 버전과 연결되어야 한다.
- 검증 실패 결과는 실패 사유와 함께 기록되어야 한다.
- 사용자 신고로 접수된 AI 응답은 평가 셋 개선에 활용할 수 있어야 한다.
- 검증 통과 산출물만 사용자에게 노출되어야 한다.

### 운영 보강 요구사항

- 시연 전 평가 셋은 최소 10개 본문을 포함해야 한다. 본문은 서사, 교훈, 시가, 예언, 복음서, 서신 등 서로 다른 유형이 섞이도록 구성한다.
- 사용자 신고로 접수된 AI 응답과 운영자가 발견한 실패 사례는 평가 셋 후보로 등록하고, 확정 여부를 운영자가 판단한다.
- 관리자 콘솔에는 검증 실패율, 반려 사유 분포, 재생성 횟수, 차단된 Q&A 유형을 확인할 수 있는 모니터링 화면을 제공한다.
- 검증 체크리스트가 변경되면 버전을 올리고, 이후 생성되는 산출물에는 변경된 버전을 연결한다.

### 수용 기준

- 검증을 통과하지 못한 해설은 사용자에게 노출되지 않아야 한다.
- 금지 표현이 포함된 산출물은 반려되어야 한다.
- AI 산출물마다 생성 시 사용된 지시 자산 버전과 검증 체크리스트 버전을 확인할 수 있어야 한다.
- 사용자 신고가 접수된 AI 응답은 운영자가 확인할 수 있어야 한다.

### 예외 처리

- 검증 시스템 장애 시 신규 산출물은 사용자 노출 상태로 전환하지 않고 검토 대기 또는 검증 실패 상태로 보류한다.
- 검증 체크리스트 버전 정보가 누락된 산출물은 사용자에게 노출하지 않는다.

## 6.15 F-15 — 사실 기반 Q&A AI

### 기능 설명

사용자는 성경 본문을 읽다가 모르는 단어, 시대상, 역사적 사실을 질문할 수 있다. AI는 단발성으로만 응답하며 이전 질문 맥락을 유지하지 않는다.

### 답변 범위

| 허용 | 차단 |
|---|---|
| 역사적 사실 설명 | 가치 판단 질문 |
| 시대상·배경 설명 | 신앙 상태 평가 |
| 단어 뜻 설명 | 자율적 묵상 권유 |
| 주석·해설 기반 보조 설명 | 단정적 설교 |
| 본문 이해를 돕는 객관 정보 | 사용자의 선택을 대신 결정하는 답변 |

### 요구사항

- AI 버튼은 성경 화면과 QT 화면에서 접근 가능해야 한다.
- AI 응답은 단발성으로 제공되어야 한다.
- 시스템은 이전 질문 맥락을 유지하지 않아야 한다.
- 가치 판단형 질문은 답변하지 않고 제한 안내를 표시해야 한다.
- 응답 하단에는 출처 또는 근거 표시가 있어야 한다.
- 사용자별 사용량 제한이 적용되어야 한다.
- 동일·유사 질문에 대한 비용 절감을 위해 캐싱 정책을 적용할 수 있다.
- AI Q&A 로그 저장 여부, 보존 기간, 학습 사용 여부는 개인정보 정책과 운영 정책에서 명시해야 한다.

### 가드레일 및 운영 요구사항

- F-15의 시스템 프롬프트, 응답 스키마, 차단 규칙, 출처 표기 방식은 별도 프롬프트·가드레일 문서에서 관리한다.
- 가드레일 프롬프트는 추상적·주관적 질문, 가치 판단 질문, 신앙 평가 질문, 단정적 설교 요청을 차단해야 한다.
- 차단 안내 메시지는 화면과 로그에서 동일한 표준 문구를 사용한다.
- 동일하거나 유사한 허용 질문은 캐싱할 수 있으나, 캐싱된 응답도 검증 기준과 출처 표기 기준을 충족해야 한다.

### 차단 안내 예시

> 이 기능은 단어, 시대상, 역사적 사실을 설명하는 용도로만 제공됩니다. 신앙적 판단이나 개인의 선택을 대신 결정하는 질문에는 답변하지 않습니다.

### 수용 기준

- 사용자가 단어 뜻을 질문하면 짧은 사실 기반 답변이 제공되어야 한다.
- 사용자가 가치 판단형 질문을 입력하면 답변 대신 제한 안내가 표시되어야 한다.
- AI 응답은 이전 질문 내용을 전제로 이어지지 않아야 한다.
- 응답에는 출처 또는 근거가 표시되어야 한다.
- 사용량 한도를 초과하면 한도 초과 안내가 표시되어야 한다.

### 예외 처리

- AI 응답 생성에 실패하면 재시도 안내를 표시한다.
- 자동 검증을 통과하지 못한 응답은 사용자에게 노출하지 않고 제한 안내를 표시한다.
- 외부 AI 서비스 장애 시 기능 일시 사용 불가 안내를 표시한다.

## 6.16 F-16 — 자유 노트

### 기능 설명

사용자는 오늘의 QT 외에도 자유롭게 노트를 작성할 수 있다. 자유 노트는 설교 노트, 기도제목 노트, 회개 노트, 감사일기로 구분한다.

### 노트 카테고리

| 카테고리 | 기반 본문 | 주요 사용 케이스 | 제목 입력 |
|---|---|---|---|
| QT 노트 | 오늘의 QT 본문 | 그날의 QT 묵상 | 선택 |
| 설교 노트 | 자유 성경 본문 | 설교를 들으며 해당 구절 기록 | 가능 |
| 기도제목 노트 | 본문 좌표 무관 | 개인·교회·이웃 기도제목 기록 | 가능 |
| 회개 노트 | 본문 좌표 무관 | 개인 회개 기록 | 가능 |
| 감사일기 | 본문 좌표 무관 | 일상 감사 기록 | 가능 |

### 요구사항

- 설교 노트는 한 절 또는 여러 절을 선택할 수 있어야 한다.
- 기도제목 노트, 회개 노트, 감사일기는 본문 좌표 없이 작성할 수 있어야 한다.
- 자유 노트는 일자별 중복 제한을 적용하지 않는다.
- 자유 노트도 기본 비공개로 생성된다.
- 사용자가 공유를 선택하면 F-10의 닉네임 공유 정책을 따른다.
- 자유 본문 좌표가 오늘의 QT 사전 생성 범위와 일치할 경우에만 해설을 노출할 수 있다.
- 자유 본문 좌표가 QT 범위와 일치하지 않으면 “준비된 해설 없음”으로 표시한다.
- 시뮬레이터는 자유 노트에서 제공하지 않는다.

### 카테고리별 입력값

| 카테고리 | 필수 입력 | 선택 입력 |
|---|---|---|
| 설교 노트 | 제목 또는 본문 내용 중 하나, 성경 좌표 | 메모, 공유 설정 |
| 기도제목 노트 | 본문 내용 | 제목, 공유 설정 |
| 회개 노트 | 본문 내용 | 제목, 공유 설정 |
| 감사일기 | 본문 내용 | 제목, 공유 설정 |

### 수용 기준

- 사용자는 자유 노트 카테고리를 선택해 노트를 작성할 수 있어야 한다.
- 설교 노트는 성경 구절을 선택할 수 있어야 한다.
- 기도제목·회개·감사일기는 성경 구절 없이 저장할 수 있어야 한다.
- 자유 노트는 같은 날짜에 여러 건 작성할 수 있어야 한다.
- 자유 노트에서는 시뮬레이터 버튼이 표시되지 않아야 한다.

### 예외 처리

- 설교 노트의 성경 좌표 조회에 실패하면 저장을 막고 구절 선택 실패 안내를 표시한다.
- 자유 노트 저장 실패 시 작성 내용이 저장되지 않았음을 명확히 표시하고 재시도 동선을 제공한다.
- 공유 전환 처리에 실패하면 공유본을 생성하지 않고 기존 비공개 상태를 유지한다.

## 6.17 F-17 — TTS

### 기능 설명

사용자는 본문과 해설을 음성으로 재생할 수 있다. TTS는 운전 중 청취, 시각 피로 완화, 접근성 보조를 목적으로 제공한다.

### 적용 범위

| 콘텐츠 | 적용 여부 |
|---|---|
| QT 본문 | 적용 |
| 일반 성경 본문 | 적용 |
| 해설 | 적용 |
| QT 노트 | 미적용 |
| 자유 노트 | 미적용 |
| 닉네임 나눔 게시글 | 미적용 |

### 요구사항

- 사용자는 본문 또는 해설 재생을 시작·일시정지할 수 있어야 한다.
- 사용자는 이전 절·다음 절로 이동할 수 있어야 한다.
- 한글과 영어 본문에 적절한 음성 언어가 적용되어야 한다.
- 음성 속도와 음성 선택 UI를 제공할 수 있다.
- 백그라운드 재생, 잠금화면 미디어 컨트롤, 블루투스 핸들러 대응은 운전 중 청취와 접근성을 고려하여 MVP 포함 권장 항목으로 분류한다. 단, OS 정책과 일정 리스크를 검토한 뒤 최종 적용 범위를 확정한다.

### 수용 기준

- 사용자가 본문 재생 버튼을 누르면 음성 재생이 시작되어야 한다.
- 사용자는 재생 중 일시정지할 수 있어야 한다.
- 사용자는 다음 절과 이전 절로 이동할 수 있어야 한다.
- 노트와 게시글 화면에는 TTS 기능이 제공되지 않아야 한다.

### 예외 처리

- 기기에서 TTS가 지원되지 않으면 기능 사용 불가 안내를 표시한다.
- 음성 데이터 로딩에 실패하면 재시도 안내를 표시한다.

## 14. v3 주요 수정 사항 요약

- 요구사항과 회의 배경을 분리하고, 본문을 요구사항 중심으로 정리하였다.
- “묵상 노트”의 의미를 QT 노트, 자유 노트, 묵상 기록으로 구분하였다.
- 공유 시 닉네임 노출 안내 요구사항을 추가하였다.
- 일반 사용자, 관리자, 자체 제작자, 시스템 배치의 권한을 분리하였다.
- 검증용 한국어 주석 자료의 접근 권한을 명확히 하였다.
- 사실 기반 Q&A AI의 실시간 검증 방식과 사후 인간 검토 방식을 구분하였다.
- QT 본문 범위 정보와 성경 본문 데이터셋의 관계를 명확히 하였다.
- 기능별 수용 기준과 예외 처리 기준을 추가하였다.
- 노트, 공유글, 댓글, 신고, 알림, AI Q&A 로그의 데이터 처리 정책을 보강하였다.
- 화면별 필수 구성 요소를 추가하였다.
- “슬라이프” 등 표현 오류를 정리하고 문체를 공식 요구사항 문서에 맞게 다듬었다.

## 15. v3 → v3.1 추가 보강 사항 요약

- 기능 목록에 리스크 수준과 주요 후속 문서를 추가하여 요구사항 추적성과 설계 연계성을 강화하였다.
- AI 산출물의 생성 지시 자산 버전뿐 아니라 검증 체크리스트 버전까지 추적하도록 보강하였다.
- 평가 셋 최소 기준을 10개 본문으로 상향하고, 사용자 신고와 검증 실패 사례를 평가 셋 후보로 환류하는 절차를 추가하였다.
- 역할별 접근 제어 매트릭스를 추가하여 일반 사용자, 관리자, 자체 제작자, 시스템 배치의 권한 경계를 명확히 하였다.
- 감사 로그 대상에 AI Q&A 차단 사례, 자동 검증 실패, 평가 셋 반영, 검증 체크리스트 버전 변경을 추가하였다.
- 공유본 스냅샷을 변경 불가 데이터로 정의하여 원본 수정·삭제와 공유본의 관계를 명확히 하였다.
- AI Q&A 로그는 저장 최소화 원칙, 익명화 또는 미저장 옵션 검토, 암호화, 접근 권한 명시 대상으로 보강하였다.
- 네트워크 오프라인 상황에서 로컬 캐시 본문을 우선 표시하는 예외 처리 기준을 추가하였다.
- 오류·권한 없음·준비 중·네트워크 실패·검증 실패 안내를 공통 오류·예외 컴포넌트로 정의하였다.
- 후속 문서를 최우선, 높음, 중간, 후속으로 구분하여 설계 진행 순서를 명확히 하였다.
- 사용자 경험 성공 기준에 초심자 사용자 테스트 80% 이상 긍정 응답 목표를 추가하였다.
