# 로그인 ERD 설명

이 문서는 QT-AI의 로그인과 회원 식별에 관련된 ERD를 초보자도 이해할 수 있도록 풀어 쓴 설명이다.

---

## 1. 로그인에서 핵심으로 보는 테이블

로그인 기본 흐름에서 핵심 테이블은 다음 2개다.

```text
members
member_auth_providers
```

역할은 이렇게 나뉜다.

| 테이블 | 쉬운 의미 |
| --- | --- |
| `members` | 우리 앱의 회원 정보 |
| `member_auth_providers` | 카카오 같은 외부 로그인 계정과 우리 앱 회원을 연결하는 정보 |

즉, `members`는 "우리 앱 안에서 이 사람이 누구인가"를 저장하고, `member_auth_providers`는 "이 사람이 어떤 카카오 계정으로 로그인했는가"를 저장한다.

---

## 2. members — 회원

`members`는 앱에 가입한 사용자 1명을 나타내는 테이블이다.

```text
members = 앱 회원 명부
```

예시 데이터:

```text
id: 1
nickname: grace2026
role: USER
status: ACTIVE
tutorial_completed_at: 2026-05-17 10:30:00
withdrawn_at: NULL
legal_retention_until: NULL
created_at: 2026-05-17 10:00:00
updated_at: 2026-05-17 10:30:00
```

### 주요 컬럼

| 컬럼 | 설명 |
| --- | --- |
| `id` | 회원 고유 번호. 다른 테이블에서 회원을 연결할 때 사용한다. |
| `nickname` | 나눔 공간에 노출되는 닉네임. 중복되면 안 된다. |
| `role` | 권한. 일반 사용자, 관리자, 시스템 배치를 구분한다. |
| `status` | 회원 상태. 정상, 탈퇴, 정지 등을 구분한다. |
| `tutorial_completed_at` | 첫 실행 튜토리얼을 완료한 시각. |
| `withdrawn_at` | 탈퇴한 시각. 탈퇴하지 않았으면 비어 있다. |
| `legal_retention_until` | 탈퇴 후 법적 보존이 끝나는 예정일. |
| `created_at` | 회원 데이터가 처음 만들어진 시각. |
| `updated_at` | 회원 데이터가 마지막으로 수정된 시각. |

### role 값

| 값 | 의미 |
| --- | --- |
| `USER` | 일반 사용자 |
| `ADMIN` | 관리자 |
| `SYSTEM` | 배치 작업이나 시스템 작업 주체 |

### status 값

| 값 | 의미 |
| --- | --- |
| `ACTIVE` | 정상 회원 |
| `WITHDRAWN` | 탈퇴 회원 |
| `SUSPENDED` | 정지 회원 |

---

## 3. member_auth_providers — 외부 로그인 연결

`member_auth_providers`는 카카오 로그인 계정과 우리 앱 회원을 연결하는 테이블이다.

```text
member_auth_providers = 외부 로그인 계정 연결표
```

예시 데이터:

```text
id: 10
member_id: 1
provider: KAKAO
provider_user_id: 3829102931
connected_at: 2026-05-17 10:00:00
created_at: 2026-05-17 10:00:00
updated_at: NULL
```

이 데이터의 뜻은 다음과 같다.

```text
우리 앱의 1번 회원은
카카오 사용자 ID 3829102931 계정으로 로그인한다.
```

### 주요 컬럼

| 컬럼 | 설명 |
| --- | --- |
| `id` | 로그인 연결 정보의 고유 번호 |
| `member_id` | 우리 앱 회원 ID. `members.id`와 연결된다. |
| `provider` | 로그인 제공자. MVP에서는 `KAKAO`를 사용한다. |
| `provider_user_id` | 카카오가 알려준 사용자 고유 ID |
| `connected_at` | 이 로그인 방식이 연결된 시각 |
| `created_at` | 데이터 생성 시각 |
| `updated_at` | 데이터 수정 시각 |

---

## 4. 왜 테이블을 2개로 나누는가?

처음에는 `members`에 카카오 ID를 바로 넣어도 될 것처럼 보인다.

```text
members
- id
- nickname
- kakao_user_id
```

하지만 나중에 구글 로그인이나 애플 로그인을 추가하면 컬럼이 계속 늘어난다.

```text
members
- kakao_user_id
- google_user_id
- apple_user_id
```

또 한 명의 회원이 여러 로그인 방식을 연결할 수도 있다.

```text
1번 회원
- 카카오 로그인
- 애플 로그인
```

그래서 외부 로그인 연결 정보는 `member_auth_providers`로 분리하는 편이 깔끔하다.

---

## 5. 카카오 로그인 흐름

카카오 로그인은 보통 이렇게 흘러간다.

```text
1. 사용자가 카카오 로그인 버튼을 누른다.
2. 카카오가 사용자 고유 ID를 알려준다.
3. 우리 서버가 member_auth_providers에서 찾는다.
4. 이미 있으면 연결된 member_id로 members를 조회한다.
5. 없으면 새 회원을 만들고, 카카오 연결 정보도 저장한다.
6. 로그인 완료 처리한다.
```

기존 회원 로그인:

```text
KAKAO + provider_user_id = 3829102931 검색
→ member_auth_providers에서 발견
→ member_id = 1
→ members.id = 1 조회
→ 로그인 성공
```

신규 회원 가입:

```text
KAKAO + provider_user_id = 9999999999 검색
→ 없음
→ members 새로 생성
→ member_auth_providers 새로 생성
→ 로그인 성공
```

## 6. UNIQUE 제약의 의미

`member_auth_providers`에는 아래 제약이 있다.

```text
UNIQUE (provider, provider_user_id)
```

뜻은 다음과 같다.

```text
같은 카카오 계정은 우리 앱에서 한 명의 회원에게만 연결될 수 있다.
```

예를 들어 아래는 가능하다.

```text
KAKAO + 3829102931 → member_id 1
```

하지만 아래처럼 같은 카카오 계정이 두 회원에게 연결되면 안 된다.

```text
KAKAO + 3829102931 → member_id 1
KAKAO + 3829102931 → member_id 2
```

그래서 `UNIQUE (provider, provider_user_id)`가 필요하다.

---

## 7. 자동 로그인까지 생각하면 필요한 추가 테이블

현재 ERD의 로그인 기본 구조는 `members`, `member_auth_providers` 두 테이블이다.

하지만 요구사항에는 자동 로그인이 있다. 자동 로그인과 로그아웃, 강제 로그아웃, 토큰 재발급까지 제대로 관리하려면 아래 테이블을 추가로 고려할 수 있다.

```text
auth_refresh_tokens
```

예상 구조:

| 컬럼 | 설명 |
| --- | --- |
| `id` | 토큰 ID |
| `member_id` | 회원 ID |
| `token_hash` | refresh token 원문이 아니라 해시값 |
| `expires_at` | 만료 시각 |
| `revoked_at` | 로그아웃이나 강제 만료 시각 |
| `created_at` | 생성 시각 |

토큰 원문을 DB에 그대로 저장하지 않고 `token_hash`로 저장하는 이유는 보안 때문이다.

---

## 8. 한 줄 요약

```text
members는 우리 앱 회원 정보,
member_auth_providers는 카카오 계정과 우리 앱 회원을 연결하는 정보다.
```

로그인 기본 구현은 이 두 테이블로 이해하면 된다.

자동 로그인과 토큰 관리를 정교하게 하려면 `auth_refresh_tokens` 테이블을 추가로 검토한다.
