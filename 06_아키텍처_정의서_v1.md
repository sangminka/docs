## 1. 전체 시스템 레이어 구조

QT-AI 백엔드는 모바일/웹 클라이언트가 호출하는 REST API 서버를 기준으로 설계한다. 화면 문서의 모든 사용자 기능은 로그인 후 접근을 기본으로 하며, API 명세의 `/api/v1`, `/api/v1/admin`, `/api/v1/system`, `/oauth2` 경계를 그대로 따른다.

```text
Client App / Web
  │
  │ HTTPS + Authorization: Bearer {accessToken}
  ▼
Nginx / Reverse Proxy
  │
  ▼
Spring Boot App
  ├─ Security Filter
  │   ├─ Access Token 인증
  │   ├─ Refresh Token 재발급
  │   ├─ Admin Role 검사
  │   └─ System Token 검사
  │
  ├─ Controller Layer
  │   ├─ /api/v1
  │   ├─ /api/v1/admin
  │   ├─ /api/v1/system
  │   └─ /oauth2
  │
  ├─ Service Layer
  │   ├─ 도메인 유스케이스
  │   ├─ 권한/소유자 검증
  │   ├─ 상태 전이 검증
  │   └─ 트랜잭션 경계
  │
  ├─ Repository Layer
  │   ├─ Spring Data JPA
  │   ├─ QueryDSL 또는 명시적 JPQL
  │   └─ 도메인별 DB 접근
  │
  ├─ External Layer
  │   ├─ Kakao OAuth
  │   ├─ LLM Provider
  │   └─ Object Storage 또는 임시 검증 자료 저장소
  │
  ├─ Batch/System Layer
  │   ├─ AI 사전 생성
  │   ├─ AI 검증 로그 생성
  │   ├─ 평가 셋 환류
  │   └─ 만료/정리 작업
  │
  ├─ Redis
  │   ├─ Refresh Token 또는 세션성 토큰 저장
  │   ├─ 단기 인증 상태
  │   └─ 사용량 제한/중복 요청 방지 캐시
  │
  └─ MySQL 8.0
      ├─ ERD v2.0 기준 전체 테이블
      ├─ 사용자/노트/나눔/알림
      ├─ 성경/QT/해설/용어
      └─ AI 생성/검증/감사 로그
```

### 1.1 레이어 의존 방향

| 레이어 | 허용 의존 |
|---|---|
| Controller | Service, Request/Response DTO, 인증 Principal |
| Service | Repository, Domain Entity, Domain DTO, External Client, Common Utility |
| Repository | Entity, Spring Data JPA, QueryDSL |
| Entity | 같은 도메인의 enum/value object, `common.entity.BaseEntity` |
| External | WebClient, 외부 요청/응답 DTO |
| Common | 도메인에 의존하지 않음 |

Controller가 Repository를 직접 호출하거나, Repository가 DTO를 반환하는 구조는 금지한다. 단순 조회 최적화가 필요하면 해당 도메인의 `query` 패키지에 읽기 전용 QueryRepository를 둔다.

## 2. 기술 스택

| 영역 | 기술 | 버전/방식 | 기준 |
|---|---|---|---|
| Runtime | JDK | 21 LTS | Spring Boot 3.x 기준 |
| Framework | Spring Boot | 3.x | REST API 서버 |
| Security | Spring Security | 6.x | OAuth2 + Bearer Token |
| OAuth | Kakao OAuth2 | 우선 적용 | API 명세 F-04 |
| DB | MySQL | 8.0 | ERD v2.0 기준 |
| ORM | Spring Data JPA | Hibernate | Entity 중심 구현 |
| Query | QueryDSL 또는 JPQL | 복잡 조회에 한정 | 목록/검색/통계 |
| Cache/Token Store | Redis | 7.x | Refresh Token, 사용량 제한, Idempotency |
| HTTP Client | Spring WebClient | 외부 AI/Kakao 호출 | `external/**`에서만 사용 |
| Validation | Jakarta Bean Validation | `@Valid`, `@Validated` | 모든 외부 입력 |
| Build | Gradle | Kotlin DSL 또는 Groovy DSL | 팀 기준 통일 |
| Test | JUnit 5, AssertJ, Mockito, MockMvc | 계층별 테스트 |
| API Docs | Spring REST Docs 또는 OpenAPI | 택1 | API 명세와 동기화 |
| Batch | Spring Scheduler 또는 Spring Batch | MVP는 Scheduler 허용 | AI 사전 생성/검증 |
| Deploy | Docker Compose + Nginx | MVP 기준 | 운영 확장 가능 |
| Monitoring | Spring Actuator | health, info, metrics | 배포/시연 안정성 |

> 본 문서는 REST API 중심으로 작성한다. 화면 템플릿 서버 사이드 렌더링은 기본 범위가 아니며, 관리자 화면을 서버 템플릿으로 만들 경우에도 API/Service/Repository 경계는 동일하게 유지한다.

### 3.1 도메인 내부 표준 구조

각 도메인은 아래 하위 패키지를 기본으로 둔다.

```text
domain/{module}
├── api              # Controller
├── application      # Service, UseCase
├── dao              # Repository, QueryRepository
├── domain           # Entity, Enum, Domain method
├── dto              # Request/Response DTO
└── exception        # 도메인 전용 예외
```

도메인 간 공유 DTO를 만들지 않는다. 여러 도메인이 같은 응답 조각을 필요로 하면 `common.dto`가 아니라 실제 소유 도메인에 DTO를 두고, 다른 도메인은 Service 메서드를 통해 값을 얻는다.

## 4. 기능 모듈별 책임 범위

### 4.1 Member/Auth 모듈

**책임**
- 카카오 OAuth 로그인, 토큰 발급/재발급/로그아웃
- 회원 프로필, 닉네임 중복 확인, 튜토리얼 완료
- 회원 탈퇴와 `WITHDRAWN`, `SUSPENDED`, `ACTIVE` 상태 처리
- 관리자 계정 여부 조회

**금지**
- 다른 도메인의 개인 데이터 삭제를 직접 수행하지 않는다.
- 탈퇴 시 노트/나눔/알림/AI Q&A 후처리는 각 소유 도메인 Service를 호출해 위임한다.

### 4.2 Bible/QT 모듈

**책임**
- 성경 권/장/절 조회
- 오늘의 QT 조회와 QT 상세 조회
- 관리자 QT 등록/수정/게시/숨김
- QT 범위와 절 매핑 검증

**금지**
- 성경 원본 JSON을 Controller에서 직접 읽지 않는다.
- QT 조회 API에서 해설/노트/시뮬레이터 세부 로직을 직접 구현하지 않는다. 필요한 요약 정보만 소유 도메인 Service를 통해 조합한다.

### 4.3 Study Content 모듈

**책임**
- 승인된 해설, 요약, 용어 풀이 조회
- `verse_explanations.APPROVED`, `glossary_terms.APPROVED` 노출 필터링
- 생성용 주석 출처와 원문 메타 관리
- 검증용 주석 원문 노출 방지 정책 반영

**금지**
- `PENDING`, `REJECTED`, `HIDDEN` AI 산출물을 사용자 API로 반환하지 않는다.
- 검증용 한국어 주석 원문을 사용자/일반 관리자 응답에 포함하지 않는다.

### 4.4 Note 모듈

**책임**
- QT 노트와 자유 노트 생성/수정/삭제/조회
- `MEDITATION`, `SERMON`, `PRAYER`, `REPENTANCE`, `GRATITUDE` 카테고리 처리
- QT 노트 1일 1건 활성 제약
- 임시저장(`DRAFT`)과 저장 확정(`SAVED`) 상태 전이
- 마이페이지 묵상 달력과 노트 집계 데이터 제공

**금지**
- 자동 저장을 구현하지 않는다.
- 공유글 스냅샷을 직접 수정하지 않는다. 공유는 Sharing 모듈에 위임한다.

### 4.5 Sharing/Community 모듈

**책임**
- 닉네임 나눔 피드와 상세 조회
- 공유글 스냅샷 생성
- 좋아요, 댓글, 공유글 숨김/삭제
- 공유 당시 닉네임/본문/구절 스냅샷 유지

**금지**
- 원본 노트를 수정하거나 삭제하지 않는다.
- 원본 노트 수정 후 공유글 본문을 자동 동기화하지 않는다.

### 4.6 Report/Notification 모듈

**책임**
- 게시글, 댓글, AI Q&A, AI 산출물 신고 접수
- 신고 처리 상태 전이
- 처리 결과 알림 생성
- 좋아요/댓글/공지/신고 결과 알림 조회와 읽음 처리

**금지**
- 신고 대상 도메인의 상태를 Repository로 직접 변경하지 않는다. 대상 도메인 Service를 통해 처리한다.

### 4.7 Praise/Mission/MyPage 모듈

**책임**
- 운영자 찬양 큐레이션 메타데이터 관리
- 사용자 내 찬양 목록 관리
- 마이페이지 대시보드에 필요한 통계/미션/알림/나눔 요약 조합
- 서버에 가사/음원 파일 저장 금지 정책 반영

**금지**
- 사용자 디바이스 음원을 서버에 업로드하는 API를 만들지 않는다.

### 4.8 AI 모듈

**책임**
- AI 사전 생성 작업, 산출물 등록, 검증 로그 등록
- 사실 기반 Q&A 요청/응답/차단/검증
- AI 산출물 승인/반려/숨김/재생성 요청
- 검증 체크리스트 버전, 평가 셋, 평가 케이스 관리
- 시뮬레이터 클립 및 컴포넌트 라이브러리 버전 추적

**금지**
- 다중 턴 채팅 세션을 만들지 않는다.
- 가치 판단, 신앙 상태 평가, 선택 대행 질문에 답변하지 않는다.
- 검증 실패 응답을 사용자에게 노출하지 않는다.

### 4.9 Admin/Audit 모듈

**책임**
- 관리자 대시보드 조회
- 관리자 세부 권한별 메뉴/처리 가능 범위 제공
- 감사 로그 조회
- 운영 변경 작업의 감사 로그 기록 표준 제공

**금지**
- 감사 로그 수정/삭제 API를 만들지 않는다.

## 5. 팀원별 소유 영역

실명 확정 전까지 문서상 소유자는 역할명으로 표시한다. 실제 팀원 이름이 정해지면 이 표만 갱신하고 패키지 구조는 변경하지 않는다.

| 팀원 | 소유 패키지 | 주요 기능 | 주요 API |
|---|---|---|---|
| 팀원 1: Backend Lead | `config`, `common`, `security`, `external`, 아키텍처 리뷰 | 공통 응답, 예외, 인증 필터, 토큰, 외부 클라이언트, 코드 규칙 | `/oauth2/*`, 토큰 공통, 공통 예외 |
| 팀원 2: Auth/Member 담당 | `domain.member`, `security.oauth` 일부 | 카카오 로그인, 회원, 프로필, 튜토리얼, 탈퇴 | `/api/v1/me/*`, `/api/v1/members/*` |
| 팀원 3: Bible/QT/Study 담당 | `domain.bible`, `domain.qt`, `domain.study` | 성경 조회, 오늘 QT, 해설/용어 조회, 관리자 QT | `/api/v1/bible/*`, `/api/v1/qt/*`, `/api/v1/admin/qt-passages` |
| 팀원 4: Note/MyPage 담당 | `domain.note`, `domain.mission` | QT 노트, 자유 노트, 묵상 달력, 대시보드 일부 | `/api/v1/notes/*`, `/api/v1/me/dashboard`, `/api/v1/me/meditation-calendar` |
| 팀원 5: Sharing/Report/Notification 담당 | `domain.sharing`, `domain.report`, `domain.notification` | 나눔, 댓글, 좋아요, 신고, 알림, 공지 알림 | `/api/v1/sharing-posts/*`, `/api/v1/reports`, `/api/v1/notifications` |
| 팀원 6: AI/Admin/Ops 담당 | `domain.ai`, `domain.admin`, `domain.audit`, `domain.praise`, `batch` | AI Q&A, 검증, 평가 셋, 관리자, 감사, 찬양, 배치 | `/api/v1/ai/*`, `/api/v1/admin/*`, `/api/v1/system/*`, `/api/v1/praise-songs/*` |

### 5.1 협업 규칙

- 자기 소유 패키지 밖의 파일은 수정 전 소유자에게 공유한다.
- `common`, `security`, `config`, `BaseEntity`, `ErrorCode`, 인증 Principal은 Lead 승인 없이 수정하지 않는다.
- Entity 컬럼, enum 값, 상태 전이, API URL 변경은 ERD/API 문서와 함께 수정한다.
- 도메인 간 직접 Repository 호출은 금지한다. 반드시 상대 도메인의 Service 또는 Facade를 호출한다.

## 6. 공통 레이어와 변경 금지 영역

### 6.1 Lead 단독 소유 영역

| 영역 | 경로 | 변경 기준 |
|---|---|---|
| 공통 응답 | `common.dto.ApiResponse`, `PageResponse`, `ErrorResponse` | API 명세 1.4~1.6과 불일치 금지 |
| 공통 예외 | `common.exception.*` | 신규 `ErrorCode` 추가는 가능하나 기존 코드명 변경 금지 |
| 인증/인가 | `security.*`, `config.SecurityConfig` | 엔드포인트 권한 변경 시 API 명세 갱신 필요 |
| 공통 Entity | `common.entity.BaseEntity` | 컬럼 변경 금지, 필요 시 ERD 동시 수정 |
| 외부 클라이언트 | `external.*` | API Key/Secret 로깅 금지 |
| 트랜잭션 정책 | Service 계층 AOP 설정 | Repository 트랜잭션 남발 금지 |

### 6.2 변경 금지 원칙

- API Prefix `/api/v1`, `/api/v1/admin`, `/api/v1/system`, `/oauth2`는 변경하지 않는다.
- ERD의 테이블명과 핵심 enum 값은 임의 변경하지 않는다.
- 공통 응답 envelope의 `success`, `data`, `error`, `timestamp`, `traceId` 필드는 유지한다.
- `204 No Content`는 예외적으로 envelope를 사용하지 않는다.
- `audit_logs`는 수정/삭제 기능을 만들지 않는다.
- 검증용 한국어 주석 원문은 사용자 API와 일반 관리자 API에서 응답하지 않는다.

## 7. 인증/인가 구조

### 7.1 인증 흐름

```text
1. Client → GET /oauth2/authorization/kakao
2. Kakao 인증 완료
3. Client/Kakao → GET /oauth2/callback/kakao
4. 서버가 provider_user_id로 member_auth_providers 조회
5. 신규 회원이면 members, member_auth_providers 생성
6. 서버 자체 accessToken, refreshToken 발급
7. Client는 Authorization Bearer accessToken으로 API 호출
8. accessToken 만료 시 POST /api/v1/auth/token/refresh
```

### 7.2 토큰 저장 정책

| 토큰 | 저장 위치 | 정책 |
|---|---|---|
| Access Token | 클라이언트 | 짧은 만료 시간 |
| Refresh Token | HttpOnly Cookie 또는 Redis 서버 저장 | MVP는 HttpOnly Cookie + Redis 저장 권장 |
| System Token | 서버 간 인증용 | `service_accounts`와 매핑 |

Refresh Token 폐기, 로그아웃, 탈퇴, 제재 시 Redis의 활성 토큰을 무효화한다.

### 7.3 인가 기준

| API 구분 | 인증 | 인가 |
|---|---|---|
| `/oauth2/**` | 불필요 | 공개 |
| `/api/v1/me/session` | 선택 | 토큰 있으면 회원 상태 반환 |
| `/api/v1/**` | USER | 본인 리소스는 `memberId` 소유자 검증 |
| `/api/v1/admin/**` | ADMIN | `members.role=ADMIN` + `admin_users.admin_role` |
| `/api/v1/system/**` | SYSTEM_BATCH | `service_accounts` 기반 서버 간 인증 |

### 7.4 관리자 세부 권한

| Admin Role | 허용 범위 |
|---|---|
| `OPERATOR` | QT 관리, 신고 처리, 공지 발행, 회원 제재, 찬양 관리 |
| `REVIEWER` | AI 산출물 검증, 체크리스트, 평가 케이스 검토 |
| `CONTENT_CREATOR` | 검증용 자료와 콘텐츠 제작 메타 관리 |
| `SUPER_ADMIN` | 전체 관리자 기능 |

`ADMIN` role만으로 모든 관리자 작업을 허용하지 않는다. 반드시 `admin_users.admin_role`을 함께 검사한다.

## 9. DTO / Entity / Repository / Service / Controller 작성 규칙

### 9.1 DTO 규칙

| 구분 | 규칙 |
|---|---|
| Request DTO | `dto.request` 또는 `dto` 하위에 `*Request`로 작성 |
| Response DTO | `*Response`로 작성 |
| Page 응답 | `PageResponse<T>`로 감싼다 |
| Validation | Request 필드에 `@NotNull`, `@NotBlank`, `@Size`, `@Pattern` 명시 |
| Entity 노출 | Entity를 Controller 응답으로 직접 반환 금지 |
| 날짜/시간 | API 응답은 ISO-8601 문자열, 서버 기준 시간대는 Asia/Seoul |

예시:

```java
public record UpdateProfileRequest(
    @NotBlank
    @Size(min = 2, max = 30)
    String nickname
) {
}
```

### 9.2 Entity 규칙

- Entity는 `domain/{module}/domain`에 둔다.
- 모든 운영 테이블 Entity는 `BaseEntity`를 상속한다.
- 상태 값은 문자열 상수 대신 enum을 사용한다.
- Entity 연관관계는 필요한 방향만 건다. 모든 FK를 양방향으로 만들지 않는다.
- `@ManyToOne(fetch = FetchType.LAZY)`를 기본으로 한다.
- 컬렉션은 외부에서 직접 수정하지 않고 도메인 메서드를 통해 변경한다.
- 소프트 딜리트 대상은 `delete()` 같은 도메인 메서드에서 상태와 `deletedAt`, `activeUniqueKey`를 함께 처리한다.

### 9.3 Repository 규칙

- Repository는 `dao` 패키지에 둔다.
- 기본 CRUD는 `JpaRepository<Entity, Long>`를 사용한다.
- 복잡한 목록/검색/통계는 `*QueryRepository`로 분리한다.
- Repository 메서드명 쿼리가 길어지면 JPQL 또는 QueryDSL로 전환한다.
- Repository는 Entity 또는 primitive 값을 반환한다. API Response DTO를 반환하지 않는다.
- 다형 참조(`reports.target_type + target_id`, `audit_logs.target_type + target_id`)는 애플리케이션 Service에서 무결성을 검증한다.

### 9.4 Service 규칙

- Service는 `application` 패키지에 둔다.
- 외부에서 호출되는 유스케이스 메서드는 명확한 동사로 시작한다.
- 쓰기 Service 메서드에는 `@Transactional`을 붙인다.
- 조회 Service 메서드에는 `@Transactional(readOnly = true)`를 붙인다.
- 권한 검증, 소유자 검증, 상태 전이는 Service에서 수행한다.
- 다른 도메인의 DB를 직접 수정하지 않고 해당 도메인 Service를 호출한다.

### 9.5 Controller 규칙

- Controller는 `api` 패키지에 둔다.
- URL은 API 명세의 경로와 완전히 일치해야 한다.
- Controller는 인증 Principal 추출, Request 검증, Service 호출, Response 매핑만 수행한다.
- 비즈니스 조건문을 Controller에 두지 않는다.
- 성공 응답은 `ApiResponse.success(data)`로 반환한다.
- 생성은 `201 Created`, 삭제/로그아웃/완료 처리는 필요 시 `204 No Content`를 사용한다.

## 11. 트랜잭션 경계

### 11.1 기본 원칙

- 트랜잭션은 Service 공개 메서드에 둔다.
- Controller와 Repository에는 원칙적으로 트랜잭션을 두지 않는다.
- 단순 조회는 `@Transactional(readOnly = true)`를 사용한다.
- 외부 API 호출은 DB 트랜잭션 내부에서 오래 잡지 않는다.
- 외부 API 호출 결과를 저장해야 하면 `요청 저장 -> 외부 호출 -> 결과 저장` 순서를 검토하고, 실패 시 재시도 가능 상태를 남긴다.

### 11.2 주요 유스케이스별 트랜잭션

| 유스케이스 | 트랜잭션 경계 | 주의 |
|---|---|---|
| 회원 가입/로그인 | `AuthService.handleOAuthCallback` | `members`, `member_auth_providers` 원자성 |
| 프로필 수정 | `MemberService.updateProfile` | 닉네임 중복 예외 처리 |
| QT 등록/수정 | `QtAdminService.savePassage` | 날짜 중복, 절 범위 검증 |
| QT 조회 | readOnly | 본문/노트 요약 조회 최적화 |
| 학습 콘텐츠 조회 | readOnly | `APPROVED`만 조회 |
| 노트 생성/수정 | `NoteService.saveNote` | QT 노트 활성 1건 제약, `activeUniqueKey` |
| 노트 공유 | `SharingService.shareNote` | 원본 노트 조회 + 공유 스냅샷 생성 원자성 |
| 좋아요 | `SharingReactionService.like` | 중복 좋아요 방지 |
| 신고 처리 | `ReportAdminService.resolve` | 신고 상태, 대상 상태, 알림, 감사 로그 |
| 알림 읽음 | `NotificationService.markAsRead` | 본인 알림만 변경 |
| AI Q&A 요청 | `AiQaService.createRequest` | 차단/실패 상태를 반드시 저장 |
| AI 산출물 승인 | `AiAssetReviewService.approve` | 산출물 상태 + 노출본 연결 + 감사 로그 |
| 체크리스트 활성화 | `AiChecklistService.activate` | 기존 ACTIVE retired 처리 |
| 탈퇴 | `MemberWithdrawalService.withdraw` | 회원 상태 변경 + 도메인별 후처리 |

### 11.3 잠금/동시성

- QT 노트 1건 제약은 DB 유니크 제약과 Service 검증을 함께 사용한다.
- 좋아요 중복은 DB 유니크 제약을 우선한다.
- 체크리스트 활성화처럼 단일 ACTIVE가 필요한 작업은 비관적 잠금 또는 유니크 제약을 검토한다.
- `Idempotency-Key`가 있는 생성 API는 Redis에 요청 키를 저장해 중복 생성을 막는다.

### 12.1 테스트 유형

| 유형 | 대상 | 도구 | 작성 기준 |
|---|---|---|---|
| 단위 테스트 | 도메인 메서드, 상태 전이 | JUnit 5, AssertJ | enum/state 변경 필수 |
| Service 테스트 | 유스케이스, 트랜잭션 | SpringBootTest 또는 DataJpaTest 조합 | 소유자/권한/상태 검증 |
| Repository 테스트 | 쿼리, 인덱스성 조회 | DataJpaTest | 목록/검색/통계 |
| Controller 테스트 | API 계약 | MockMvc | 공통 envelope와 HTTP status 검증 |
| Security 테스트 | 인증/인가 | MockMvc + 테스트 Principal | USER/ADMIN/SYSTEM 경계 |
| External 테스트 | Kakao/LLM client | MockWebServer | 타임아웃/실패 응답 |

### 12.2 필수 테스트 시나리오

- 토큰 만료 시 `TOKEN_EXPIRED` 반환.
- 일반 사용자가 관리자 API 접근 시 `FORBIDDEN` 반환.
- `ADMIN`이지만 `admin_users.admin_role`이 없으면 관리자 세부 기능 접근 불가.
- `APPROVED`가 아닌 해설/시뮬레이터/AI 산출물은 사용자 API에서 미노출.
- 같은 회원+QT+MEDITATION 활성 노트 중복 생성 불가.
- 공유글 생성 후 원본 노트 수정 시 공유 스냅샷이 바뀌지 않음.
- AI 가치 판단 질문은 `AI_POLICY_BLOCKED` 처리.
- 신고 처리 완료 시 알림과 감사 로그가 생성됨.

## 13. 개발 시 주의사항

### 13.1 요구사항 정합성

- 로그인 없는 기능 화면은 만들지 않는다.
- 오늘의 QT가 로그인 후 중심 진입점이다.
- 요약/해설/용어/시뮬레이터/TTS/AI는 사용자 선택 시 제공한다.
- 노트는 기본 비공개다.
- 자동 저장은 구현하지 않는다.
- 공유는 닉네임 노출 안내 후 수행한다.
- 공유글은 공유 시점 스냅샷이며 원본과 자동 동기화하지 않는다.

### 13.2 ERD 정합성

- `notes`는 QT 노트와 자유 노트를 함께 담는다.
- QT 노트는 `category=MEDITATION`, `qt_passage_id`를 가진다.
- 설교 노트는 `note_verses`를 사용한다.
- `reports.target_type + target_id`, `audit_logs.target_type + target_id`는 다형 참조이므로 Service에서 대상 존재 여부를 확인한다.
- `ai_generated_assets`와 사용자 노출 테이블(`verse_explanations`, `simulator_clips`)은 분리한다.
- `validation_reference_jobs`의 원문/색인은 임시 사용 후 만료/삭제 상태를 관리한다.

### 13.3 API 정합성

- URL, HTTP Method, 권한은 `05_API_명세서_v1.md`를 기준으로 한다.
- `204 No Content` 외에는 공통 envelope를 사용한다.
- 페이징은 API 명세의 `content`, `page`, `size`, `totalElements`, `totalPages`, `first`, `last`, `sort` 형식을 따른다.
- ErrorCode 이름은 프론트와 QA fixture에 영향을 주므로 임의 변경하지 않는다.

### 13.4 보안/개인정보

- Access Token, Refresh Token, Kakao Token, LLM API Key를 로그에 남기지 않는다.
- 사용자 노트 본문과 AI Q&A 원문 로그는 필요한 범위만 저장한다.
- 탈퇴 후 복구 불가 정책을 훼손하지 않는다.
- 검증용 한국어 주석 원문은 일반 관리자에게도 노출하지 않는다.
- 찬양 가사와 음원 파일은 서버에 저장하지 않는다.

### 13.5 성능

- 오늘 QT 첫 응답 목표는 P95 500ms 이하다.
- 묵상 달력 조회 목표는 P95 300ms 이하다.
- 사실 기반 Q&A 목표는 P95 5초 이하다.
- 목록 API는 기본 `size=20`으로 시작한다.
- `TEXT`, `JSON` 컬럼 조건 검색은 기본 금지이며 필요 시 별도 검색 전략을 수립한다.
